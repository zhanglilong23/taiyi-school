# 太一九星 · 使用指南（GUIDE）

## 快速开始

太一九星是角色化 AI 协作流程。你可以按流程一步步走，也可以只召唤某颗星处理特定问题。

## 两种使用模式

### 模式一：手动模式（默认）

每颗星完成后都会给出明确的下一步召唤指令。你按提示手动 `@下一颗星` 即可。

**优点**：
- 每步可控，适合需要人工把关的复杂任务
- 不依赖环境是否支持 skill 间互调
- 任何时候可以改派、暂停、插入新任务

**示例**：

```
你：@天枢 我想做一个商城系统
天枢：需求真言已产出，下一步请手动 @天璇 判 Epic + 设计
你：@天璇
天璇：设计包已产出，下一步请手动 @天玑 拆任务
...
```

### 模式二：司衡自动模式（可选）

如果你的运行环境支持 skill 间互调，可以召唤 `@司衡` 让司衡自动推进流程。

```
你：@司衡 继续
司衡：当前应执行下一步：@天玑。正在自动唤醒...
```

**自动唤醒失败时**（环境不支持），司衡会一次性降级为手动提示：

```
司衡：当前应执行下一步：@天玑。当前环境不支持自动唤醒，请手动召唤。
```

## 九星召唤速查

| 你想做什么 | 召唤 |
|---|---|
| 理清模糊需求 | @天枢 |
| 做技术/产品设计 | @天璇 |
| 拆可执行任务 | @天玑 |
| 按规格写代码（新功能）| @天权 |
| 修 bug / 性能优化 | @开阳 |
| 编码后自检 | @玉衡 |
| 审查代码 / 独立 review | @洞明 |
| 诊断 bug 根因 | @瑶光 |
| 非功能性风险扫描 | @隐元 |
| 自动推进流程 | @司衡（环境支持时）|

## 常见场景

### 场景 1：从零做一个新功能

```
@天枢 → @天璇 → @天玑 → @天权 → @玉衡 → @洞明
```

### 场景 2：修一个线上 bug

```
@瑶光 查根因 → @开阳 修复 → @玉衡 自检 → @洞明 审查
```

### 场景 3：只做代码审查

```
@洞明 审查这个 PR
```

洞明是工具入口，不需要经过前面的星。

### 场景 4：流程被打回

```
@洞明 审查 → 打回 → 手动 @瑶光 查根因 → 按根因 @天权/@天璇/@天玑/@天枢
```

## 状态文件

两个状态文件是九星与司衡之间的唯一接口：

- `.taiyi/_workspace/status.md`：当前星状态、下一步星、手动召唤指令
- `.taiyi/_workspace/queue.md`：当前 Epic 的 Task 队列

你可以随时问 `@司衡 现在什么状态` 查看当前进度。

## 司衡不是必须的

如果环境不支持自动唤醒，或者你更喜欢手动控制，完全可以不用 `@司衡`。每颗星都会告诉你下一步该 `@` 谁。

司衡只是可选的自动路由增强层。九星 SKILL 内部不依赖司衡存活。

## 提示语标准

每颗星完成后会输出：

```markdown
【太一流转】
- 当前完成：{星名} 已完成 {产出物/动作}
- 下一步：{next_action} → 请手动 {manual_invoke}
- 前置状态：{ready / blocked / 等待用户输入}
- （可选）环境支持自动流转时：也可 @司衡 继续
```

## 注意事项

1. **玉衡是必经节点**：天权/开阳编码后必须经过玉衡自检，才能进入洞明审查。
2. **洞明前检查玉衡自检摘要**：洞明会检查 `impl.md` 是否含玉衡自检摘要，没有会拒绝审查。
3. **瑶光是诊断星**：瑶光完成后 `next_star=IDLE`，会按根因给出具体 `@X星` 建议。
4. **司衡只读不写**：司衡不会修改任何文件，只根据状态文件给出路由建议或自动唤醒。
