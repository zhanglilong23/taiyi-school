## 一、五衰定性与鉴影

- 🔍 **观势鉴影**：九星职责看似分明，实则存在**两处决策真空**——
  1. **输入侧真空**：客户输入模糊或跨角色时，LLM 缺少排他性判定规则，陷入"做 / 拒 / 转"的纠结；
  2. **出口侧真空**：角色完成后，默认下一步不清晰，内化驱动星（玉衡、隐元）与司衡五指规则断层，LLM 陷入"自己做 / 交别人 / 跳过某步"的纠结。
- 🔍 **寻脉鉴影**：问题不是角色太多，而是**角色之间的切换条件没有形成决策树**，导致每个星都在做开放题而非选择题。
- **关键脉络**：
  - `CONTEXT.md` 主链路写明：天枢 → 天璇 → 天玑 → 天权/开阳 → **玉衡（必经）** → 洞明；
  - 但 `skills/tianquan/SKILL.md:222` 与 `skills/kaiyang/SKILL.md:139` 的"完成后"直接写 `下一步=洞明审查`，**跳过了玉衡**；
  - 司衡五指调度规则（`skills/siheng/SKILL.md:128-138`）只有指极/指营/指校/指通/指回，**没有"指衡"**，玉衡由天权/开阳"内化驱动"；
  - 天权/开阳 SKILL 又在"手动模式"中提供"跳过玉衡"选项，与 CONTEXT.md 的"必经"直接冲突。

## 二、病位与阵眼

### 病位 A：输入侧缺少"请求性质判定树"

- **位置**：`skills/siheng/SKILL.md` 第一层意图分类 + 各星 SKILL 输入自审
- **症状**：
  - 司衡遇到"多星弱匹配"时只给出"问用户这听起来像 X 或 Y"，没有给出按输入模糊度/任务类型的快速判定树；
  - 天枢虽有"混沌判定"，但当用户直接 @天璇 / @天玑 / @天权 一个模糊请求时，各星只能依赖 LLM 自己判断 R2 错误@使用；
  - 没有统一问题："这个请求是否模糊？是否跨角色？是否属于本星？"
- **五衰之相**：**虚浮相** —— 每个星都写了大量 When NOT to use 和邻接表，但没有给出一个让 LLM 快速做选择题的极简判定树。
- **凶险程度**：**P1 重要修缮**
- **影响面**：每次用户输入模糊或跨角色时，LLM 消耗额外 token 纠结，容易越界或误拒。

### 病位 B：玉衡是"必经"但流程图中被跳过

- **位置**：
  - `docs/CONTEXT.md:26` 与 `:34` 写明玉衡是必经；
  - `skills/tianquan/SKILL.md:222` / `skills/kaiyang/SKILL.md:139` 写 `下一步=洞明审查`；
  - `skills/siheng/SKILL.md:128-138` 五指规则无"指衡"。
- **症状**：
  - 天权/开阳编码完成后，status.md 的下一步直接指向洞明；
  - 玉衡被降级为"内化驱动"，是否执行取决于天权/开阳能否自动唤醒玉衡；
  - 天权/开阳 SKILL 明确说"手动模式跳过玉衡是降级……体系推荐三层但容忍两层"。
- **五衰之相**：**朽骨相** —— 流程骨架（CONTEXT.md）与实现细节（各星 SKILL）不一致，残留历史妥协。
- **凶险程度**：**P1 重要修缮**
- **影响面**："必经"玉衡可被动跳过，独立审查层失效，洞明接收未经系统自检的代码。

### 病位 C：司衡五指规则未覆盖内化驱动星

- **位置**：`skills/siheng/SKILL.md` 五指调度规则
- **症状**：
  - 玉衡、隐元不在司衡五指规则中；
  - 玉衡由天权/开阳驱动，隐元由洞明驱动；
  - 一旦进入手动/降级模式，用户/LLM 不知道何时 @玉衡、何时 @隐元。
- **五衰之相**：**幻翳相** —— 司衡被称为"调度中枢"，但实际未调度所有关键节点，产生"完整调度"的虚假印象。
- **凶险程度**：**P1 重要修缮**
- **影响面**：手动模式下流程断点，角色内耗加剧。

### 病位 D：各星"完成后"的下一步写法不一致

- **位置**：各星 SKILL.md "完成后"章节
- **症状**：
  - 天权/开阳：直接写 `下一步=洞明审查`；
  - 玉衡：自检通过 → 天权/开阳 commit，自检打回 → 天权/开阳自修；
  - 洞明：放行 → 取下个 Task / 归档，打回 → @瑶光；
  - 瑶光：主动召唤 → 待用户决定，被动触发 → 人工调度执行；
  - 隐元：下一步=洞明综合决策。
- **五衰之相**：**死水相** —— 每个星各自为政描述出口，缺乏统一的"单点出口"原则。
- **凶险程度**：**P1 重要修缮**
- **影响面**：LLM 完成一项任务后需要自行选择下一步，容易跳过必要节点或重复调度。

### 病位 E：隐元结果回归洞明的闭环不清晰

- **位置**：`skills/dongming/SKILL.md:159-171` 隐元驱动 + `skills/yinyuan/SKILL.md:98-103` 完成后
- **症状**：
  - 洞明"尝试 @隐元 唤醒"，失败时让用户选择"洞明自行兼顾风险维度"；
  - 隐元产出 risk.md 后"交回洞明综合决策"，但洞明 SKILL 没有说明如何在审查报告中融合隐元结果；
  - 洞明已经开始审查，隐元是中途插入，没有明确的"隐元结果必须阻塞/通过"规则。
- **五衰之相**：**幻翳相** —— 隐元看起来是专项审计，但实际是否影响放行由洞明"自行兼顾"决定。
- **凶险程度**：**P2 炼化建议**
- **影响面**：高安全/高并发任务的非功能性风险审查可能被稀释。

## 四、破局法诀

### ⚠️ P1 重要修缮

1. **在司衡 SKILL 中增加"请求性质判定树"**
   - 落点：`skills/siheng/SKILL.md` 第一层意图分类。
   - 建议增加一个显式决策树：
     ```
     用户请求进入司衡：
       1. 是元问题（状态/Epic 列表）？→ 司衡直答
       2. 输入模糊/混沌/跨角色？→ 天枢
       3. 已明确需求但无设计？→ 天璇
       4. 有设计包但无任务契约？→ 天玑
       5. 有契约需编码？→ 天权/开阳（按 mode）
       6. 代码已编码需自检？→ 玉衡
       7. 已 commit 需审查？→ 洞明
       8. 审查打回需查根因？→ 瑶光
       9. 审查中发现非默认风险？→ 隐元
       10. 仍不确定 / 多星弱匹配？→ 问用户一个具体问题
     ```
   - 让 LLM 做选择题，而不是开放题。

2. **统一"单点出口"原则**
   - 每个星完成后只能有一个默认下一步，除非用户显式改派。
   - 建议统一为：
     | 当前星 | 默认下一步 |
     |--------|-----------|
     | 天枢 | 天璇 |
     | 天璇 | 天玑 |
     | 天玑 | 天权/开阳（按 mode）|
     | 天权/开阳 | 玉衡 |
     | 玉衡 | 洞明 |
     | 洞明（放行）| 下一个 Task 或 Epic 归档 |
     | 洞明（打回）| 瑶光 |
     | 瑶光 | 源头星（或用户决策）|
     | 隐元 | 洞明 |

3. **修正玉衡"必经"与流程图的冲突**
   - 删除 `skills/tianquan/SKILL.md:222` 与 `skills/kaiyang/SKILL.md:139` 中 `下一步=洞明审查`，改为 `下一步=玉衡自检`。
   - 删除天权/开阳 SKILL 中"手动模式可跳过玉衡"的选项，改为：
     - 自动模式：天权/开阳唤醒玉衡；
     - 手动模式：告知用户"请手动 @玉衡 自检，再继续 @洞明 审查"。
   - 玉衡不通过 → 不允许进入洞明。

4. **扩展司衡五指规则为七星调度，或明确司衡只调主线**
   - 方案 A（推荐）：增加"指衡"规则，让司衡也调度玉衡；隐元保持由洞明按需触发，但结果必须回归洞明。
   - 方案 B：明确司衡只调主线星（天枢→天璇→天玑→天权/开阳→玉衡→洞明），玉衡/隐元等内化驱动星不在司衡调度范围，但要在文档中显式标注。
   - 无论选哪个方案，都要在 SKILL 中写清，不能既有"调度中枢"之名，又无完整调度之实。

### 💡 P2 炼化建议

1. **为每个星增加"3 问自审"快速判定清单**
   - 开工前强制自答：
     1. 这个请求属于我职责范围吗？（是/否/不确定）
     2. 我的前置输入存在且明确吗？（R1 检查）
     3. 如果不属于我，应该交给哪个星？（查邻接表）
   - 将"不确定"统一路由到司衡或用户澄清，减少 LLM 自行猜测。

2. **明确隐元结果如何影响洞明放行**
   - 在洞明 SKILL 中增加规则：
     - 隐元发现 P0/P1 风险 → 洞明必须打回；
     - 隐元发现 P2 风险 → 登记 intervention，可放行但需跟踪；
     - 隐元未触发或未发现 → 洞明按功能性审查继续。

3. **增加流程图的一致性校验测试**
   - 在 `tests/` 中增加一个测试，校验：
     - 每个星的"完成后"下一步是否指向合法星；
     - 主线星的下一步是否形成闭环；
     - CONTEXT.md 主链路与各星 SKILL 的"完成后"是否一致。

### 📜 契约补全（建议司南/天璇纳入）

- **新增约束条款**："九星主线流程必须形成单点出口：每个星完成后默认下一步唯一，除非用户显式改派。"
- **新增约束条款**："玉衡是必经节点，任何情况下不得从 天权/开阳 直接跳到 洞明。"
- **新增约束条款**："司衡必须在 SKILL 中显式声明调度的星范围（主线七星或全部），未纳入调度的内化驱动星必须标注触发条件与回归节点。"
- **新增约束条款**："各星 SKILL 的'完成后'章节必须与 CONTEXT.md 主链路一致，不一致视为契约违反。"
