## 一、问题陈述

太一学派当前九星角色协作存在内耗：
1. 客户输入模糊时，LLM 纠结"做 / 拒 / 转"；
2. 角色完成后，LLM 纠结"自己做审查 / 交其他角色 / 跳过某步"。

根因：九星 SKILL 内部混用自动/手动双模式，司衡调度范围未明确，状态流转未形成唯一接口。

## 二、边界声明

### 2.1 适用范围

- 太一学派 `skills/*` 下全部九星 SKILL.md；
- 太一学派 `skills/siheng/SKILL.md`；
- 太一学派 `.taiyi/_workspace/status.md`；
- 太一学派 `.taiyi/_workspace/queue.md`。

### 2.2 不适用范围

- 不修改九星的核心职责定义（天枢判需求、天璇设计、天玑拆任务、天权/开阳编码、玉衡自检、洞明审查、隐元风险、瑶光诊断）；
- 不修改 `.taiyi/` 产物层目录结构；
- 不引入新的角色或新的流程阶段；
- **不修改 `skills/*/references/` 下的模板文件**——这些模板由 `SKILL-AUTHORING.md` 规范（技能自包含、多星共用模板各存副本、`test-references.sh` 守护一致性），不在本契约范围内。本契约修改 SKILL.md 时，若涉及模板引用，需确保路径仍符合 `SKILL-AUTHORING.md` 规范。

## 三、方案：九星手动模式 + 司衡可选自动路由层

### 3.1 核心原则

1. **九星 SKILL 内部只按手动模式设计**；
2. **每个星完成后更新 `_workspace/status.md` 并输出标准提示语**；
3. **司衡是可选的自动路由增强层**：在支持 skill 间互调的环境中，司衡读取状态文件后代为用户召唤下一个星；在不支持的环境中，司衡只给出手动召唤建议；
4. **状态文件是九星与司衡之间的唯一接口**。

### 3.2 九星 SKILL 的约束

#### 3.2.1 九星 SKILL 正文不依赖司衡

**定义**：
- **正文**：SKILL.md 中描述角色职责、输入自审、处理流程、纪律约束等规则性内容（不包括"完成后"向用户输出的提示语和验证命令）。
- **完成后提示语**：每个星执行完毕向用户输出的标准提示，属于用户交互文本，可提及司衡作为可选入口。

**约束**：
- 禁止在正文中写"司衡会自动唤醒你""等待司衡调度""司衡负责调度到下一步"等依赖；
- 禁止在正文中写 `skill("siheng")` 或 `skill("司衡")` 等调用；
- 允许在"完成后"提示语中把司衡作为可选入口提及，例如："下一步：@天玑（或 @司衡 自动继续）"。

#### 3.2.2 每个星完成后必须执行的标准动作

1. 更新 `.taiyi/_workspace/status.md`；
2. 向用户输出【太一流转】标准提示语。

### 3.3 `_workspace/status.md` 流转字段标准

**格式**：保留现有 Markdown 表格格式，在现有字段基础上扩展列。不强制改为 YAML frontmatter，以避免破坏现有 SKILL 的写入逻辑。

扩展后的字段：

| 字段 | 含义 | 必填 | 示例 |
|------|------|------|------|
| `current_star` | 当前完成或正在执行的星 | 是 | 天璇 |
| `current_star_status` | 当前星状态 | 是 | done / running / rejected / blocked / waiting_user |
| `next_star` | 默认下一步星（单点出口） | 是 | 天玑 |
| `next_action` | 下一步动作的人类可读描述 | 是 | 拆任务 |
| `manual_invoke` | 手动模式下用户应输入的召唤指令 | 是 | @天玑 |
| `block_reason` | REJECTED/BLOCKED/WAITING_USER 时必填 | 条件必填 | R1 + 缺 design.md + 建议路由: 天璇 |

**状态枚举**：`IDLE / RUNNING / DONE / REJECTED / BLOCKED / WAITING_USER`（与当前 `status.md` 一致）。

#### 3.3.1 状态转换规则

| 当前星 | current_star_status | next_star | 说明 |
|--------|---------------------|-----------|------|
| 天枢 | done | 天璇 | 需求真言产出后进入设计 |
| 天璇 | done | 天玑 | 设计包产出后进入拆任务 |
| 天玑 | done | 天权/开阳 | 按 ready Task 的 mode 字段分流 |
| 天权 | done | 玉衡 | 编码后必经自检 |
| 开阳 | done | 玉衡 | 编码后必经自检 |
| 玉衡 | done | 洞明 | 自检通过后进入审查 |
| 玉衡 | rejected | 天权/开阳 | 自检不通过，自修 |
| 洞明 | done | 天权/开阳 / 瑶光 / IDLE | 放行后：有 ready Task → 继续编码；无 ready Task 但需归档 → 瑶光观势；全部完成 → IDLE |
| 洞明 | rejected | 瑶光 | 打回后查根因 |
| 瑶光 | done | IDLE | 诊断完成后根因可能指向多星，无法单点出口；在 `block_reason` 中给出建议路由，在 `manual_invoke` 中给出具体召唤指令 |
| 隐元 | done | 洞明 | 风险扫描结果回归洞明综合决策 |

**洞明 done 后的分支判定**：
1. 读 `queue.md`，有 ready Task → `next_star=天权/开阳`（按首个 ready Task 的 mode）；
2. 无 ready Task 但 Epic 未完成归档 → `next_star=瑶光`（触发 Epic 级架构回归）；
3. Epic 已归档 → `current_star=IDLE`，`next_star=IDLE`。

**特殊星的 next_star 处理**：

- **瑶光**：诊断星，输出是"根因路由建议"而非固定流程下一步。完成后 `next_star=IDLE`，`manual_invoke` 按根因填充为具体 `@X星`，`block_reason` 写明建议路由。司衡读到 `next_star=IDLE` 时输出："诊断已完成，建议下一步：{manual_invoke}。理由：{block_reason}。"
- **隐元**：由洞明触发、回归洞明的专项扫描星。完成后 `next_star=洞明`，但洞明已处于审查流程中，此处的"next_star=洞明"表示结果回归而非新唤醒。

#### 3.3.2 异常/分支状态

- **R1 输入不符**：`current_star_status=REJECTED`，`block_reason="R1 + 缺 XXX + 建议路由: YY星"`；
- **R2 错误@使用**：`current_star_status=REJECTED`，`block_reason="R2 + 请求错配 + 建议路由: YY星"`；
- **R3 能力边界**：`current_star_status=BLOCKED`，`block_reason="R3 + 卡因 + 建议升级用户/回天璇"`；
- **WAITING_USER**：`current_star_status=WAITING_USER`，`block_reason="需用户澄清: 具体问题"`。

### 3.4 九星 SKILL 完成后标准提示语

每个星完成后必须输出以下格式：

```markdown
【太一流转】
- 当前完成：{星名} 已完成 {产出物/动作}
- 下一步：{next_action} → 请手动 {manual_invoke}
- 前置状态：{ready / blocked / 等待用户输入}
- （可选）环境支持自动流转时：也可 @司衡 继续
```

示例（天璇完成后）：

```markdown
【太一流转】
- 当前完成：天璇已产出 design.md
- 下一步：拆任务 → 请手动 @天玑
- 前置状态：ready
- 环境支持自动流转时：也可 @司衡 继续
```

### 3.5 司衡的行为

#### 3.5.1 司衡的输入

- 用户请求；
- `.taiyi/_workspace/status.md`；
- `.taiyi/_workspace/queue.md`（用于任务队列）。

#### 3.5.2 司衡的输出

- 自动模式下：尝试唤醒 `next_star`；
- 手动模式下：输出"请手动 {manual_invoke}"；
- 元问题：直接回答状态/Epic/任务信息。

#### 3.5.3 Agent 能力判定方法

司衡**不预先判定**当前 Agent 是否支持 skill 间互调，而是采用**运行时探测**：

1. 每次需要自动唤醒时，尝试调用 `skill("{next_star}")`；
2. 调用成功 → 继续观察目标星是否按预期更新 `status.md`；
3. 调用失败或目标星未按预期更新 → 触发降级。

不维护"当前 Agent 是否支持自动唤醒"的状态变量，避免状态与实际能力脱节。

#### 3.5.4 自动唤醒失败判定标准

满足以下任一条件即判定为失败：

1. **调用异常**：`skill("{next_star}")` 抛出异常、返回错误、或提示 not found；
2. **无响应**：调用后超过合理时间（如 30 秒）目标星未开始执行；
3. **状态未更新**：调用后目标星未按预期更新 `status.md` 的 `current_star` 字段；
4. **用户打断**：用户在自动唤醒过程中明确说"停""改派XX"。

#### 3.5.5 司衡的降级规则

```markdown
尝试自动唤醒 {next_star} 失败（当前 Agent 不支持 skill 间互调）
→ 一次性降级，不重试：
   "当前应执行下一步：{manual_invoke}。当前环境不支持自动唤醒，请手动召唤。"
```

### 3.6 玉衡、隐元的定位

#### 3.6.1 玉衡

- 在手动模式下，玉衡是可独立召唤的角色；
- 天权/开阳完成后必须提示用户"请手动 @玉衡"；
- 玉衡自检通过后，**天权/开阳**必须在 `impl.md` 末尾追加**玉衡自检摘要**，并更新 `status.md` 的 `current_star=玉衡(done)`、`next_star=洞明`；
- 洞明输入自审必须检查 `impl.md` 是否含**玉衡自检摘要**，无摘要 → R1 拒绝，路由回溯玉衡；
- 在自动模式下，司衡读取 `next_star=玉衡` 后自动唤醒玉衡，玉衡完成后自动唤醒洞明。

**玉衡自检摘要标准格式**：

必须出现在 `impl.md` 末尾，作为最后一个 Markdown 章节。

```markdown
## 玉衡自检摘要

- 自检时间：{YYYY-MM-DD HH:MM}
- 自检结果：通过 / 不通过

### 契约 DONE 逐条对照

| # | DONE 条目 | 验证方式 | 结果 | 证据 |
|---|----------|---------|------|------|
| 1 | {DONE-1 描述} | {命令/操作} | 通过 | {命令输出摘要或截图说明} |
| 2 | {DONE-2 描述} | {命令/操作} | 通过 | {命令输出摘要或截图说明} |

### 白名单与越界检查

- 改动文件列表：{git diff --name-only 输出}
- 是否全部在白名单内：是 / 否
- 越界文件（如有）：{文件路径}

### 边界与异常覆盖

- 正常流：已验证 / 未验证
- 异常流 1：{异常场景} → {处理策略} → 已验证 / 未验证
- 异常流 2：...
```

**判定规则**：洞明用以下命令检查 impl.md 是否含玉衡自检摘要：

```bash
grep -q "^## 玉衡自检摘要$" reports/TASK-*/impl.md
```

无命中 → R1 拒绝："缺玉衡自检摘要，请先 @玉衡 自检"。

**玉衡-洞明过渡保障机制**：

```markdown
1. 玉衡自检通过 → 天权/开阳在 impl.md 追加【玉衡自检摘要】
2. 天权/开阳更新 status.md：current_star=玉衡(done), next_star=洞明
3. 洞明开工前自审：impl.md 含【玉衡自检摘要】？
   - 是 → 进入审查
   - 否 → R1 拒绝："缺玉衡自检摘要，请先 @玉衡 自检"
```

#### 3.6.2 隐元

- 隐元由洞明在审查时按需触发（手动 @隐元 或自动唤醒）；
- 隐元产出 risk.md 后回归洞明；
- 洞明 SKILL 必须明确隐元结果如何影响放行（见本契约 §4.2）。

## 四、契约硬约束

### 4.1 九星 SKILL 硬约束

<HARD-GATE>
1. 九星 SKILL 正文不得依赖司衡的自动唤醒机制；
2. 每个星完成后必须更新 `_workspace/status.md` 的 `current_star / current_star_status / next_star / next_action / manual_invoke`；
3. 每个星完成后必须输出【太一流转】标准提示语；
4. `next_star` 必须是单点出口，禁止一个星完成后给出多个可选下一步；**例外：瑶光作为诊断星，完成后 `next_star=IDLE`，在 `manual_invoke` 和 `block_reason` 中给出唯一建议路由**；
5. 玉衡是必经节点，天权/开阳完成后 `next_star` 必须是玉衡，禁止直接指向洞明；
6. 玉衡自检通过后，天权/开阳必须在 `impl.md` 末尾追加**玉衡自检摘要**，否则视为未经过玉衡；
7. 洞明输入自审必须检查 `impl.md` 是否含**玉衡自检摘要**，无摘要则 R1 拒绝。
</HARD-GATE>

### 4.2 司衡 SKILL 硬约束

<HARD-GATE>
1. 司衡只读状态文件，不写任何文件；
2. 司衡自动唤醒失败后必须一次性降级，不重试、不空转；
3. 司衡在手动模式下只输出"请手动 {manual_invoke}"，不替用户决定；
4. 司衡读到 `current_star_status=REJECTED/BLOCKED` 时，按 `block_reason` 中的建议路由回溯或升级用户。
</HARD-GATE>

### 4.3 隐元影响洞明放行的规则

- 隐元发现 P0/P1 级风险 → 洞明必须打回；
- 隐元发现 P2 级风险 → 登记 intervention，洞明可放行但需跟踪；
- 隐元未发现风险 → 洞明按功能性审查继续。
