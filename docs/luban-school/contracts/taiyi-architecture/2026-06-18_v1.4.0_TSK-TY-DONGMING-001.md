## 三、STORY 拆解

### STORY-1：洞明输入自审检查玉衡自检摘要
- 洞明开工前自审：按 `grep '^## 玉衡自检摘要$'` 检查 impl.md。
- 无摘要 → R1 拒绝："缺玉衡自检摘要，请先 @玉衡 自检"。

### STORY-2：洞明增加隐元结果影响放行规则
- 隐元发现 P0/P1 级风险 → 洞明必须打回。
- 隐元发现 P2 级风险 → 登记 intervention，洞明可放行但需跟踪。
- 隐元未发现风险 → 洞明按功能性审查继续。

### STORY-3：洞明放行/打回后的 next_star 标准化
- 洞明 done 后基于 queue.md 分支判定：
  - 有 ready Task → next_star=天权/开阳
  - 无 ready Task 但 Epic 未完成归档 → next_star=瑶光
  - Epic 已归档 → current_star=IDLE, next_star=IDLE
- 洞明 rejected → next_star=瑶光

### STORY-4：瑶光 next_star=IDLE，manual_invoke 按根因填充
- 瑶光完成后 `next_star=IDLE`。
- `manual_invoke` 按根因填充为具体 `@X星`。
- `block_reason` 写明建议路由。

### STORY-5：隐元 next_star=洞明
- 隐元完成后 `next_star=洞明`，表示结果回归洞明综合决策。

### STORY-6：其余五星 next_star 标准化
- 天枢 done → next_star=天璇
- 天璇 done → next_star=天玑
- 天玑 done → next_star=天权/开阳（按 mode）

### STORY-7：CONTEXT.md 更新主链路
- 更新主链路描述，明确手动模式为默认，司衡为可选增强。
- 与各星 SKILL 的 next_star 一致。

### STORY-8：GUIDE.md 增加手动模式与司衡使用说明
- 增加"手动模式与司衡"使用说明。
- 说明九星可独立召唤，司衡在支持自动唤醒时代为调度。
