

	

## 一、学派命名
**太一学派**

> 太一藏于水，行于四时。周而又始，以己为万物母；一缺一盈，以己为万物经。 ——《太一生水》
> 太一生水，水反辅太一，天地由是而成，万象由是而序。太一藏于极，不行而万物自行，不言而契约自立——这就是架构约束思想。

```
│   天璇心法 ─── 架构决断：权衡利弊，决断技术方向                     │
```

### 3.2 为什么这样分？
| 法器       | 星宿         | 传统含义   | 工程含义                                         | 映射逻辑          |
| ---------- | ------------ | ---------- | ------------------------------------------------ | ----------------- |
| **洞真经** | 天枢（贪狼） | 掌正义是非 | **需求正判**：这条需求该不该做、优先级对不对     | 贪狼主判 → 判需求 |
|            | 天璇（巨门） | 掌决断智慧 | **架构决断**：用cron还是事件驱动、选什么技术栈   | 巨门主断 → 断架构 |
|            | 天玑（禄存） | 掌机运变化 | **机运拆解**：把Epic拆成Story/Task，捕捉每个边界 | 禄存主变 → 拆变化 |
| **洞玄经** | 天权（文曲） | 掌文运才华 | **文心编码**：写优雅、可读、可维护的代码         | 文曲主文 → 写代码 |
|            | 开阳（武曲） | 掌武勇刚毅 | **武毅攻坚**：复杂逻辑、性能优化、棘手bug修复    | 武曲主武 → 攻坚   |
|            | 玉衡（廉贞） | 掌善恶审判 | **廉明自检**：commit前自检，对错分明             | 廉贞主审 → 自审   |
| **洞神经** | 瑶光（破军） | 掌破除变革 | **破军查障**：找到bug根因，破除隐患              | 破军主破 → 破bug  |
|            | 洞明（左辅） | 掌辅弼助力 | **左辅质门**：质量门禁，辅助放行/打回            | 左辅主辅 → 辅判   |
|            | 隐元（右弼） | 掌隐秘守护 | **右弼暗守**：安全审计，守护看不见的风险         | 右弼主隐 → 守暗   |
**两个隐星归洞神经，不是随意的——校验就是"看见看不见的东西"，隐星主隐，正合此意。**

## 四、三法器详细定义
### 4.1 洞真经（规划法器）
**一句话：洞见真实，出契约。**
| 维度         | 定义                                                         |
| ------------ | ------------------------------------------------------------ |
| **输入**     | 用户原始需求                                                 |
| **输出**     | 规划包 = 需求真言 + 架构文档 + ADR + Epic/Story/Task契约全集 |
| **三重心法** | 天枢 → 天璇 → 天玑，一次出齐                                 |
```
用户需求
   │
   ▼
【天枢·需求正判】
   │ 这条需求值不值得做？优先级对不对？有没有伪需求？
   │ 有歧义 → 追问用户
   │ 输出：需求真言 + 验收标准草案
   ▼
【天璇·架构决断】
   │ 用什么架构？什么技术栈？接口怎么定？
   │ 权衡利弊，做决策，不做完美主义
   │ 输出：架构文档 + ADR + 架构约束清单 + 风险清单
   ▼
【天玑·机运拆解】
   │ 每个Epic怎么拆？Story边界在哪？异常场景有哪些？
   │ 机械执行拆解规则，不修改架构决策
   │ 输出：Story契约（填满所有字段）+ Task列表
   ▼
规划包 → 交司衡入队
```
**天枢和问道不同：问道只"澄清"，天枢还要"正判"——判定需求是否值得做、优先级是否合理。这是贪狼"掌正义与是非评判"的直接映射。**
---
### 4.2 洞玄经（执行法器）
**一句话：洞悉玄机，写代码，自检，commit。**
| 维度         | 定义                                |
| ------------ | ----------------------------------- |
| **输入**     | 单个Story契约（从司衡队列领取）     |
| **输出**     | 已commit未push的代码 + 自检报告     |
| **三重心法** | 天权/开阳（选一执行）→ 玉衡（必过） |
```
Story契约
   │
   ▼
【判断模式】
   │ Task mode = tianquan（文心编码）？
   │   → 新建模块、复杂逻辑、核心实现
   │ Task mode = kaiyang（武毅攻坚）？
   │   → 修bug、性能优化、棘手问题
   ▼
【天权·文心编码】或【开阳·武毅攻坚】
   │ 严格遵循契约的INTERFACE/SCENARIOS/CONSTRAINTS
   │ 不改白名单以外的文件
   ▼
【玉衡·廉明自检】（必过）
   │ 对照契约逐项检查：
   │   DONE → 逐条
   │   SCENARIOS → 异常覆盖
   │   CONSTRAINTS → 约束合规
   │   DON'T → 是否越界
   │ 通过 → commit（message含Story-ID + 自检摘要）
   │ 不通过 → 自修，最多2轮
   ▼
已commit代码 → 交洞神经
```
**天权和开阳的区别：**
- 天权（文曲）：写新代码，追求优雅和可读性
- 开阳（武曲）：攻坚修bug，追求快速有效
- **不是两个独立角色，是同一法器的两种模式**，由契约Task的mode字段决定
---
### 4.3 洞神经（校验法器）
**一句话：洞彻神明，验代码，过则推，不过则打回。**
| 维度         | 定义                                       |
| ------------ | ------------------------------------------ |
| **输入**     | 已commit的代码 + 对应Story契约             |
| **输出**     | 放行→push / 打回→bug清单                   |
| **三重心法** | 洞明（必过）→ 隐元（按需）→ 瑶光（打回时） |
```
已commit代码
   │
   ▼
【洞明·左辅质门】（必过）
   │ 对照契约：
   │   DONE → 逐条验证（看行为，不看代码有没有写）
   │   SCENARIOS → 场景覆盖
   │   CONSTRAINTS → 约束合规
   │   三张清单 → 功能/性能/安全基础
   │ 不通过 → 打回（附bug清单）
   ▼ 通过
【判断安全等级】
   │ Story契约 security = high？
   │
   ├─ 否 → 直接放行
   ▼ 是
【隐元·右弼暗守】（按需触发）
   │ SQL注入/XSS/命令注入
   │ 认证/授权/权限越界
   │ 敏感数据泄露
   │ 不通过 → 打回（附安全报告）
   ▼ 通过
放行 → push + 标记Story done
```
**瑶光·破军查障（打回时触发）：**
```
洞明打回
   │
   ▼
【瑶光·破军查障】
   │ 不只是"打回bug清单"
   │ 而是深入分析：
   │   根因是什么？（需求不清？架构约束不够？实现偏差？）
   │   哪些是表面bug，哪些是深层问题？
   │   是否需要回到洞真经重新评估？
   │
   │ 输出：根因分析 + 修复建议
   │ 如果根因在架构 → 触发天璇重新决断
   │ 如果根因在实现 → 打回洞玄经，附精确修复指引
   ▼
打回 → 洞玄经修复 → 洞神经再审
```
**瑶光的价值不只是"发现bug"，而是"破除根因"——破军主破，破的不是表面症状，是根因。**
---
### 4.4 司衡（调度引擎）
**一句话：**司掌权衡，契物流转。

| 规则     | 触发              | 动作                                        |
| -------- | ----------------- | ------------------------------------------- |
| **指真** | 用户提需求        | 调用洞真经 → 规划包入队                     |
| **指玄** | 队列有ready的Task | 分配给洞玄经                                |
| **指神** | 洞玄经commit      | 调用洞神经                                  |
| **指通** | 洞神经放行        | push + 分配下一个Task                       |
| **指回** | 洞神经打回        | 带bug清单+根因分析重新分配洞玄经（最多2轮） |
**司衡没有SKILL.md，它就是调度逻辑，写在项目的工作流配置里（workflow / Makefile / scripts）。**
---
### 4.5 太一（契约核心）
**太一不是一个角色，不是一个法器，是一个原则：**

> **一切围绕契约转，无契约不行动。**
> 太一体现在：
1. 洞真经输出契约 → 洞玄经按契约执行 → 洞神经按契约校验
2. 任何偏离契约的行为，必须回到洞真经重新规划
3. 契约是唯一真源
---
## 五、完整链路
### 5.1 常规链路（3步）
```
洞真经（天枢→天璇→天玑）→ 洞玄经（天权/开阳→玉衡）→ 洞神经（洞明→push）
```
### 5.2 高风险链路（4步）
```
洞真经 → 洞玄经 → 洞神经（洞明→隐元→push）
```
### 5.3 打回链路（5步）
```
洞玄经 → 洞神经（洞明打回）→ 瑶光查障 → 洞玄经（修复）→ 洞神经（再审→push）
```
### 5.4 根因在架构（回到洞真经）
```
... 洞神经打回 ...
→ 瑶光判：根因在架构
→ 回到洞真经·天璇（重新决断）
→ 天玑重新拆解受影响的Story
→ 洞玄经重新执行
```
### 5.5 Epic完成
```
Epic下所有Story done
   │
   ▼
洞神经·瑶光（架构回归）
   │ 全局审视：接口是否干净、实现是否偏离
   │ 通过 → Epic done
   │ 偏离 → 触发天璇重新评估
```

```markdown
- 文件白名单: {仅允许修改的文件路径}
- 影响模块: {模块名}
- 必须保持兼容: {接口签名、文件格式、状态机}
- 本Story可新增/修改: {新接口、新字段}
## 场景与异常（SCENARIOS）
- 正常流:
  1. {步骤}
  2. {步骤}
- 异常流:
  - {异常} → {处理策略}
  - {异常} → {处理策略}
  - {异常} → {处理策略}
## 架构与实现约束（CONSTRAINTS）
- ADR引用: {ADR-xxx: 决策内容}
- 禁止: {不得引入/不得修改}
- 必须: {必须使用的模式/工具}
## DONE（验收标准 / Acceptance Criteria）
- [ ] {场景级验收}
- [ ] {异常场景覆盖}
- [ ] {性能/安全基础要求}
## DON'T（约束边界）

## Task拆分
| Task-ID | 描述 | 心法 | 粒度 | 依赖 |

| TASK-001 | {动词：具体操作} | tianquan | ≤2h | - |
| TASK-002 | {动词：具体操作} | kaiyang | ≤30min | TASK-001 |
```

## 九、三个SKILL.md骨架
### 9.1 洞真经 SKILL.md
```markdown
# 洞真经·规划法器
## 身份
太一学派规划法器。洞见真实，出契约。
内修三重心法：天枢 → 天璇 → 天玑。
## 输入
用户原始需求（自由文本）
## 输出
规划包：
1. 需求真言 + 验收标准草案（天枢心法）
2. 架构文档 + ADR + 架构约束清单 + 风险清单（天璇心法）
3. Epic/Story/Task 契约全集（天玑心法）
## 心法·天枢（需求正判）
贪狼主正义评判。需求当前，先判是非。
触发：收到用户需求后首先执行
- 正判：这条需求值不值得做？优先级对不对？有没有伪需求？
- 澄清：有歧义则追问用户，不留模糊
- 去伪：识别XY问题——用户说想要X，真正需要的是Y
- 输出：需求真言 + 验收标准草案
- 不做技术决策
## 心法·天璇（架构决断）
巨门主决断智慧。方向当前，果断抉择。
触发：天枢心法完成后自动执行
- 模块划分 + 接口协议
- 技术选型 + ADR（架构决策记录）
- 架构约束清单（不可变接口 + 禁止项 + 不变量）
- 风险清单 + 应对策略
- 权衡利弊，做决策，不做完美主义
- 不拆Story/Task
## 心法·天玑（机运拆解）
禄存主机运变化。Epic当前，精确拆解。
触发：天璇心法完成后自动执行
- 每个Epic → N个Story（填满模板所有字段）
- 每个Story → M个Task（AI单会话原子操作）
- 每个Task标注心法模式（tianquan/kaiyang）和安全等级（normal/high）
- 任何字段填不满 = 拆解未完成
### 拆解铁律
1. 一个Story只涉及一个架构层
2. 一个Story可被一个AI编码会话完成
3. 每个Story至少3个异常流
4. Task描述必须动词开头
5. 无依赖的Task必须可并行
## 铁律
- 三重心法一次出齐，不中途返回用户确认（除非天枢有歧义需追问）
- 天玑不得修改天璇的架构决策
- 天枢不得做技术决策
- 无契约不行动（太一原则）
```
### 9.2 洞玄经 SKILL.md
```markdown
# 洞玄经·执行法器
## 身份
太一学派执行法器。洞悉玄机，写代码，自检，commit。
内修三重心法：天权/开阳 → 玉衡。
## 输入

## 心法·天权（文心编码）
文曲主文运才华。新篇当前，以文心写优雅代码。
触发：Task mode = tianquan

- 追求可读、可维护、优雅

- 不改白名单以外的文件
- 不做契约未要求的架构变更
```

### 9.3 洞神经 SKILL.md
```markdown
# 洞神经·校验法器
## 身份
太一学派校验法器。洞彻神明，验代码，过则推，不过则打回。
内修三重心法：洞明 → 隐元（按需）→ 瑶光（打回时）。
## 输入
已commit的代码 + 对应Story契约
## 输出
- 放行 → push + 标记Story done
- 打回 → bug清单 + 根因分析 + 修复建议
## 心法·洞明（左辅质门）
左辅主辅弼助力。代码当前，以辅正守质量门禁。
触发：每次commit后必须执行
- 对照契约DONE逐条验证（看行为，不看代码有没有写）
- 对照SCENARIOS验证场景覆盖
- 对照CONSTRAINTS验证约束合规
- 三张清单：功能 / 性能 / 安全基础
- 任何一项不通过 → 打回
## 心法·隐元（右弼暗守）
右弼主隐秘守护。暗处当前，以深守安全底线。
触发：Story契约 security = high
- SQL注入 / XSS / 命令注入
- 认证/授权/权限越界
- 敏感数据泄露
- 外部输入校验
- 不通过 → 打回（附安全报告）
## 心法·瑶光（破军查障）
破军主变革破除。障碍当前，以破军之力破根因。
触发：洞明或隐元打回时执行
- 不只列bug，更查根因：
  - 需求不清？→ 天枢问题
  - 架构约束不够？→ 天璇问题
  - 实现偏差？→ 天权/开阳问题
- 根因在架构 → 触发洞真经·天璇重新决断
- 根因在实现 → 打回洞玄经，附精确修复指引
- Epic完成后 → 架构回归：全局审视接口干净度、实现偏离度
## 打回规则
- 附具体bug清单（哪条DONE没过/哪个场景没覆盖/哪个约束违反）
- 附根因分析（不是表面bug，是根因）
- 最多2轮打回，第3轮升级人工
- 打回后洞玄经修复，只需验证打回项
## 铁律
- 不写代码
- 不修改契约
- 只能放行或打回，没有第三种结果
- 放行 = push权限
```
