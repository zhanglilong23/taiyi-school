## 〇、天文同构：体系结构的源头

太一学派的结构不是设计出来的，是**与天文同构**的。理解了天文，就理解了体系。

### 0.1 太一、北极星、北斗九星的真实关系

| 名称 | 本质 | 出处 |
|------|------|------|
| 北极星 | 最靠近北天极的恒星，天穹中几乎不动的星 | 天文 |
| 北辰 | 北极星的方位雅称，"北方之枢纽" | 《论语》"为政以德，譬如北辰" |
| 紫微星 | 北极星的天宫称谓，因位于紫微垣（天帝居所）| 道教星官 |
| 太一 | 北极星的神格称谓，至高天神的星象化身 | 《楚辞·东皇太一》 |

**关键事实：北极星、北辰、紫微、太一是同一颗星的四种身份，不是四颗星。**

### 0.2 北斗九星围绕北极星旋转

```
              太一（北极星，不动）
                     ★
                     ↑
          天枢天璇连线指向（指极）
                     │
        北斗九星围绕旋转（昼夜不没）
   ┌─────────────────────────────┐
   │  斗魁：天枢·天璇·天玑·天权      │
   │  斗杓：玉衡·开阳·瑶光          │
   │  隐辅：洞明（左辅）·隐元（右弼）  │
   └─────────────────────────────┘
```

- 北斗九星 = 七明星 + 二隐星（左辅洞明、右弼隐元）
- 《史记·天官书》："斗为帝车，运于中央，临制四乡。分阴阳，建四时，均五行，移节度，定诸纪。"
- 天枢（贪狼）与天璇（巨门）连线延伸五倍即指北极星——天枢别称"指极星"。

### 0.3 同构映射（不是比喻，是同构）

| 天文结构 | 体系结构 | 为什么同构 |
|---------|---------|-----------|
| 太一（北极星）不动 | **契约核心 / 不动点** | 北极星不动，契约不变——"无契约不行动" |
| 九星围绕旋转 | **九个执行节点围绕契约执行** | 斗为帝车运于中央——执行节点围绕契约运转 |
| 天枢天璇指极 | **判需求+断架构确立契约方向** | 指极=指向契约——流程起点确定北极星位置 |
| 斗运不息 | **司衡持续推进，状态机不息** | 建四时移节度——调度器驱动流转 |
| 二隐星辅弼 | **洞明/隐元按需隐现** | 隐星主隐——洞明守看得见的行为，隐元守看不见的风险，按风险标记触发 |

**这套结构与天文是一回事，不是事后映射。** 这是太一学派区别于鲁班（修仙位格是装饰）、区别于三洞真经草稿（两张皮映射）的根本所在。

## 一、太一：契约文档（不动点）

太一不是角色，不是法器，不可召唤。太一是**契约本身**——一份被九星遵循、被司衡参照的契约文档。

北极星不动，众星围绕它旋转。对应到体系——契约一旦确立即不动（神圣），九星围绕契约工作，司衡调度时以太一契约为不变参照。任何偏离契约的行为，必须回到确立契约的星（天枢/天璇）重新对准北极星。

**太一 = 契约文档，不是入口。** 用户不召唤太一（北极星不可被召唤），用户召唤的是司衡（帝车）——帝车运转，众星遵循契约工作。

```
太一（契约文档，不动）          ← 不可召唤，被遵循
        ↑
        │ 九星遵循契约工作
        │
   司衡（帝车，自动化入口）         ← @司衡 启动运转
        │
        ├─→ 天枢 → 天璇 → 天玑 → ... → 瑶光
        │    （九星按工程顺序流转）
        │
        └─→ 全程以太一契约为不变参照
```

## 二、九星：平级执行节点

**决策：九星不分组，平级围绕太一运转。** 司衡按工程顺序调度，星与星之间靠文档接力，无直接耦合。

### 2.1 九星名单（北斗九星 = 七明 + 二隐）

|----|------|------|-------|---------|------------|
| 1 | 天枢 | 贪狼·指极星 | 明 | 判需求 | 掌正义是非 |
| 2 | 天璇 | 巨门 | 明 | 断架构 | 掌决断智慧 |
| 3 | 天玑 | 禄存 | 明 | 拆任务 | 掌机运变化 |
| 4 | 天权 | 文曲 | 明 | 文心编码 | 掌文运才华 |
| 5 | 玉衡 | 廉贞 | 明 | 自检 | 掌善恶审判 |
| 6 | 开阳 | 武曲 | 明 | 武毅攻坚 | 掌武勇刚毅 |
| 7 | 瑶光 | 破军 | 明 | 查根因 | 掌破除变革 |
| 8 | 洞明 | 左辅 | 隐 | 质门守门 | 掌辅弼助力 |
| 9 | 隐元 | 右弼 | 隐 | 非功能性风险守护 | 掌隐秘守护 |

> **二隐星的工程含义**：校验就是"看见看不见的东西"，隐星主隐，正合此意。

### 2.2 各星详述

#### 天枢（贪狼 · 指极星）— 判需求

**作用**：把用户模糊的混沌表达，收敛成一句明确的需求真言。不只是澄清——还要**正判**（值不值得做？是不是伪需求？边界在哪？）。诚实的判官，不是 yes-machine。

- **输入**：混沌（用户原始表达——感觉、不满、想法）
- **输出**：需求真言（混沌原念、已确认假设、所求之道、筑基之痛、真需求、伪需求已斩、Not Doing、未明之境、用户确认）
- **门禁**：假设必浮现（不默默填空）/ 歧义清零（含糊必须追问）/ 伪需求必斩（识别 XY 问题）/ 反谄媚（需求不合理温和直说）/ 不做技术决策
- **方法论**：四步判需求（详见 2.8 节）
- **入口**：✅ 开放——需求澄清是交互性场景，用户常想"只澄清不执行"

#### 天璇（巨门）— 设计（产品+UI+架构）

**作用**：基于需求真言做完整的设计工作——产品设计、UI 原型、架构设计三项，按需求性质选择做哪几项，不强制全做。权衡利弊做决策，不做完美主义。

三项设计是同一层（设计层）的正交面，不该拆成三个角色（会制造交接断层）：
| 设计工作 | 回答什么 | 产出 |
|---------|---------|------|
| 产品设计 | 做什么功能、用户流程怎么走 | 产品规格（功能列表、用户旅程）|
| UI 原型 | 用户看到什么、怎么交互 | 界面草图/线框图/交互说明 |
| 架构设计 | 怎么实现、模块怎么分、接口怎么定 | 技术方案+ADR+约束清单 |

按需求性质选择：
- 纯后端 API 需求 → 只做架构设计
- 纯 UI 调整需求 → 只做 UI 原型
- 全栈新功能 → 三项都做

- **输入**：需求真言
- **输出**：设计包（按需含技术假设 + 选定方案+ADR + 产品规格 + UI原型 + 技术方案 + 约束清单 + 风险清单）
- **门禁**：架构决策落地为可机检约束（不能只说"用JWT"，要写"认证返回 JWT，过期2h，字段 token"）/ 不拆 Task（拆解归天玑）/ 不判需求 / 探索现有代码先行（不凭空设计）/ 提2-3方案带trade-offs（不直接给单一方案）/ 设计包自审四查（占位符/一致性/范围/歧义）
- **方法论**：四步流程 + 三项设计（详见 2.11 节）
- **入口**：✅ 开放——设计讨论是创造性对话，用户常想"只讨论方案不下定论"

#### 天玑（禄存）— 拆任务

**作用**：基于天璇的设计包，把需求拆成可执行的 Task 契约。两层结构（Task + 依赖序），不要鲁班 EPIC/STORY/SPEC 四层（太重）。每个 Task 标执行参数和风险标记。

- **输入**：天璇的设计包（产品规格/UI原型/技术方案/ADR/约束清单）
- **输出**：任务契约（每个 Task 含 TASK-ID + DONE + DON'T + mode + 风险标记 + 依赖盘点 + 文件白名单）
- **门禁**：字段填满 / DONE 可机检 / DONE 必含 E2E smoke / 依赖盘点硬门禁（未就绪拒签）
- **关键职责一：填 mode 字段**——新建类 Task（新模块、核心逻辑）→ mode=tianquan；修复/优化类 Task（bug、性能）→ mode=kaiyang
- **关键职责二：填风险标记字段**（选填，默认安全值，只标涉及的维度）：
  - `security`: normal / high（涉及外部输入/认证/敏感数据）
  - `concurrency`: none / involved（涉及并发/共享状态）
  - `resource`: none / managed（涉及资源管理：连接/句柄/内存）
  - `data`: read / mutated（是否变更持久数据）
  - `external`: none / depended（是否依赖外部服务）
  - `performance`: normal / critical（是否在热路径）
  风险标记决定隐元触发哪些维度的非功能性检查。不标=隐元不触发（简单 Task 不查）。
- **关键职责三：拆解方法论**（详见 2.6 节）
- **关键职责四：三档授权**（详见 2.7 节）
- **入口**：❌ 不开放——拆任务需明确输入（设计包），直接调无意义

#### 天权（文曲）+ 开阳（武曲）— 编码（双模式）

**作用**：按契约写代码。同一执行单元的两种模式，由契约 Task 的 mode 字段决定。

- **天权模式（mode=tianquan，文心）**：新建模块、核心逻辑、架构实现。重"理"——工整、可读、可维护、优雅。慢工出细活
- **开阳模式（mode=kaiyang，武毅）**：修 bug、性能优化、攻坚难题。重"效"——直击要害、最小改动、快速有效。兵贵神速
- **输入**：任务契约
- **输出**：commit 的代码 + 自检报告
- **前置**：开工前 5 分钟主动盘点（数据流上下游/可能失效环节/外部依赖/最大挂点/DONE 有无真实触发）
- **门禁**：只改白名单内文件 / E2E 必跑（[交付]必贴 smoke 真实输出）/ 玉衡自检通过才 commit / commit 附自检摘要 / 异常熔断（3次不同方案失败→停→阻塞报告）
- **方法论**：编码五步 + 双模式风格 + 五纪律（详见 2.12 节）
- **入口**：❌ 不开放——编码需契约输入，没有契约直接调无意义

> **为什么是双模式而非两个独立职责**：代码开发内部没有干净的正交二分（前后端/实现测试/新建修复都不正交或不普适）。文曲武曲是**同一编码动作的两种风格策略**——文重质量，武重效率。mode 由天玑拆任务时标定，不由用户选。

#### 玉衡（廉贞）— 自检

**作用**：代码写完后，对照契约逐项自检。**玉衡是开发者自测，不是 QA**（QA 是洞明）。写码者的第一道自我把关——自己先跑通，再交洞明独立审查。

- **输入**：天权/开阳产出的代码
- **输出**：自检通过（附证据）/ 自检打回报告
- **门禁**：对照 DONE 逐条 / 对照 SCENARIOS 检查异常覆盖 / 对照 CONSTRAINTS 检查约束 / 对照 DON'T 检查越界 / **发现契约外影响必须登记 intervention.md（不准只标注不登记）** / **贴证据不贴声称**（本次会话跑过的才算，"should pass"不是证据）/ 不通过→自修最多 2 轮，第 3 轮标记需洞明
- **方法论**：分层自检（详见 2.9 节）
- **入口**：❌ 不开放（内部步骤）
- **非严格启用**：默认内化进天权/开阳收尾必过步骤。重型场景可显式独立

> **玉衡 vs 洞明的标准差异**：玉衡是"我自己跑通了"（自测通过，标准低）；洞明是"我独立证明它对了"（doubt-driven 对抗审查，标准高）。差异是设计意图——玉衡放过的小问题，洞明会抓；玉衡抓不到的确认偏误，洞明的 doubt 循环破。

#### 洞明（左辅）— 质门（守门人）

**作用**：独立审查 + 放行/打回 + push/merge + 归档 + 收尾。守门人完整职责链：把关→放行→入库→落锁。**体系灵魂——独立审查，禁止信任写码者单方声称。**

- **输入**：已 commit 的代码 + 任务契约
- **输出**：放行 → push/merge + 6R 归档 + 会话收尾；打回 → 打回单（附 bug 清单 + 根因分类）
- **门禁**：独立重跑验证（禁止信任天权单方声称，自己跑 git diff + 重跑验证）/ 五张清单（证据/边界/越权/Smoke凌驾/task-queue一致性）/ Smoke 凌驾契约（没写 E2E 也要审）/ 一票否决不可委托 / push 前查 commit 规范 / 6R 归档 / **放行前清点 intervention.md open 条目——有本次产生的必须明确去向（升级Task/延后P2/用户确认），不能登记了就放**
- **前置模式：设计评审**——天玑产出契约后、天权编码前，洞明读需求真言+技术方案+契约审设计合理性（架构是否支撑需求、契约是否完整、有无遗漏场景）→ 放行进营建 / 打回回天枢天璇
- **非功能性风险守护**：洞明按契约风险标记触发隐元相应维度（security=high→安全性；concurrency/resource→鲁棒性；data=mutated→可靠性；performance=critical→性能；external=depended→兼容性）
- **入口**：✅ 开放——独立审查 PR、独立清理工作区（归档/收尾/状态重置）等场景，不走全流程

> 洞明的本职是"质门"（守门人），职责链为：**审查 → 放行/打回 → push/merge → 归档 → 收尾**。这不是五件事堆一起，是守门人"把关→放行→入库→落锁"的连贯链条。与天文意象自洽——洞明（左辅）"掌辅弼助力"，放行/归档/收尾都是辅弼助力的体现。

> **merge 冲突处理规则**：洞明在 merge 时遇到冲突，**交给天权解决，洞明不自己解**。理由——守门人绝不改代码（裁判不下场踢球）；冲突的解决需要理解两边代码意图，这是编码决策归天权。洞明可做的事：**描述冲突**（哪些文件、冲突标记位置），帮天权定位，但不动手改。例外：纯文档/格式/空白冲突（无逻辑判断）洞明可自行解决。判断标准：**冲突解决需要理解代码逻辑吗？需要→天权，不需要→洞明可解。**

#### 隐元（右弼）— 非功能性风险守护

**作用**：系统性守护"看不见但会出事"的非功能性风险。安全只是其中一种——凡是隐藏的、非行为层面的风险，皆归隐元。与天文意象自洽：右弼"掌隐秘守护"，隐元守的就是隐秘风险。

**守护五维度**（按风险信号触发，非无差别全查）：

| 维度 | 检查内容 | 触发信号 |
|------|---------|---------|
| **安全性** | 注入/越权/泄露/认证授权 | security=high |
| **鲁棒性** | 并发竞态/死锁/资源泄漏/异常吞没/边界值 | concurrency=involved 或 resource=managed |
| **可靠性** | 数据一致性/事务边界/重试幂等/降级策略 | data=mutated |
| **性能** | N+1查询/大对象/无界增长/热路径阻塞 | performance=critical |
| **兼容性** | 依赖版本/接口破坏性变更/环境假设 | external=depended |

- **输入**：代码（契约标注的风险维度）
- **输出**：非功能性风险报告（按维度分节，附具体场景）
- **门禁**：发现问题必须附具体场景证据（"并发场景下A和B竞态，因为..."）/ 只报实证不报"潜在"（不接受"建议增加鲁棒性"等模糊话）/ 轻微问题走溢出机制（登记 intervention.md，不强制当场修）/ **不直接放行——结果交洞明综合决策**（隐元是洞明的专项扫描仪，不做放行决策）
- **入口**：❌ 不开放（非功能性风险守护子流程）
- **非严格启用**：默认内化进洞明（洞明按风险标记触发相应维度）。重型场景可显式独立

**按信号触发，不无差别全查**：没有风险标记的简单 Task（纯展示组件/配置改动），隐元不触发。天玑只标涉及的维度，其余默认安全值（normal/none/read）。详见契约风险标记字段。

**与洞明的分工**：
```
洞明（左辅·质门）：审"功能性"——代码做对了没有（行为/契约/DONE）
隐元（右弼·暗守）：审"非功能性"——代码会不会暗中出事（安全/鲁棒/性能/可靠/兼容）

洞明是明面上的守门人（看得见的行为）
隐元是暗处的守护者（看不见的风险）
两颗隐星正好对应"看得见 vs 看不见"两类审查——与天文意象完全自洽。
```

**防过度设计的三约束**：
1. 按信号触发，不无差别全查（无标记不触发）
2. 只报实证，不报"潜在"（必须附具体场景）
3. 轻微问题走溢出机制（登记不阻塞，和洞明溢出机制一致）

#### 瑶光（破军）— 查根因（双面诊断）

**作用**：双面诊断——**照骨**（bug 根因诊断）+ **观势**（架构体检）。不只列 bug，而是定位"根因在哪颗星"并精准路由。破的不是表面症状，是根因。

- **输入**：打回事件（被动触发）/ 用户报障（主动召唤，@瑶光）
- **输出**：根因分析 + 路由建议（主动召唤时只出报告不续接，用户决定下一步）
- **门禁**：根因定位到具体星（需求不清→天枢 / 架构约束不够→天璇 / 契约歧义→天玑 / 实现偏差→天权开阳 / 非功能性风险→隐元）/ 没有反馈回路不猜测根因 / 修复3次失败→质疑架构非假设 / 最多 2 轮打回，第 3 轮升级用户
- **方法论**：双面诊断（详见 2.10 节）
- **Epic 完成时**：观势——架构回归，全局审视接口干净度、实现偏离度。偏离→回天璇重新评估
- **入口**：✅ 开放——debug 诊断是探索性场景，用户常想"只查根因不修复"

### 2.3 工程流转顺序（司衡按此调度）

```
天枢(判需求) → 天璇(断架构) → 天玑(拆任务)
  → [设计评审：洞明前置读三文档审合理性]
  → 天权/开阳(写码) → 玉衡(自检) → 天权/开阳 commit
  → 洞明(质门审查，按风险标记触发隐元相应维度)
  ├─ 放行 → 洞明 push/merge → 洞明 6R归档+收尾 → 司衡取下个Task
  └─ 打回 → 瑶光(查根因) → 路由回天枢/天璇/天权
```

注意：**这是工程顺序，不是分组**。九星平级，只是司衡按"想清楚→做出来→验明白"的自然工程逻辑确定调度先后。

### 2.4 入口与启用一览

| 星 | 入口 | 默认启用 | 备注 |
|----|------|---------|------|
| 天枢 | ✅ 开放 | 独立节点 | 需求澄清，直达不续接 |
| 天璇 | ✅ 开放 | 独立节点 | 架构讨论，直达不续接 |
| 天玑 | ❌ 仅司衡调度 | 独立节点 | 需技术方案输入 |
| 天权/开阳 | ❌ 仅司衡调度 | 独立节点（双模式）| 需契约输入 |
| 玉衡 | ❌ 不开放 | **内化**进天权/开阳收尾 | 显式 `@玉衡` 时独立 |
| 洞明 | ✅ 开放 | 独立节点 | 审查/独立清理工作区/归档收尾 |
| 隐元 | ❌ 不开放 | **内化**进洞明（按风险标记触发） | 显式 `@隐元` 时独立 |
| 瑶光 | ✅ 开放 | 独立节点 | debug 诊断，直达不续接 |

**主入口是 `@司衡`**（流水线，自动续接）；开放星宿（天枢/天璇/洞明/瑶光）作单星直达（不续接）。详见第七节。

### 2.5 溢出问题处理（防遗失机制）

> **解决的痛点**：鲁班学派实践中，契约有白名单限制，改动影响契约外功能时只标注不修改 → 问题遗失；审查发现的轻微问题因不影响本次验收被放过 → 同样遗失。根因是"本次契约"边界太硬，发现了契约外问题却没有强制闭环机制。

**核心原则：不擅自改（守边界），但不准遗失（强制闭环）。**

#### 三道机制

**机制一·溢出问题登记（强制义务）**

任何星发现"契约外的问题"，不擅自改，但必须登记到 `intervention.md`：

- 天权/开阳发现：改动影响了白名单外的功能
- 洞明发现：审查时发现但不影响本次验收的轻微问题
- 瑶光发现：查根因时顺带发现的其他隐患

发现契约外问题**不登记** = 玉衡自检不通过 / 洞明审查失职。登记是强制义务，不是可选动作。

登记格式：
```markdown
INT-NNN | 类型: 溢出影响/轻微问题/连带隐患 | 来源: <星名>(<场景>)
  描述: <具体问题>
  影响契约: 无（不在本次白名单/不影响本次验收）
  严重度: P0/P1/P2/待定
  状态: open
  发现时间: YYYY-MM-DD
```

**机制二·洞明放行门禁加一条——清点 intervention.md**

洞明放行前，除审本次契约，还必须清点 intervention.md 的 open 条目。有本次产生的 open 条目时，必须做以下之一才能放行：

| 去向 | 适用 | 动作 |
|------|------|------|
| 升级为新 Task | 严重（P0/P1）| 入 task-queue，本次批次或下批次处理 |
| 标记延后 | 轻微（P2）| 明确写入"待办池"，记录延后理由 |
| 用户显式确认 | 用户判断 | 用户说"我知道了，先不管"，登记用户决策 |

**不能"登记了就放"——必须有明确去向。** 这把"登记"和"闭环"连起来，问题不停在"标注了但没人管"的中间状态。

**机制三·司衡批次复看**

每次司衡开始新批次（新契约/新 Epic）前，先读 intervention.md，把历史 open 条目摆给用户：

```
司衡指极起步前：
  读 intervention.md
  若有 open 条目 → 提示用户：
    "历史遗留 N 个未闭环问题：
     INT-007 (P1): xxx
     INT-012 (P2): xxx
     本次批次是否纳入处理？或继续遗留？"
```

历史问题不会永远沉在清单里——每个新批次开始时重新审视，要么纳入处理，要么显式继续遗留（遗留也是显式决策，不是遗忘）。

#### 为什么不会导致过度设计

- **不要求修改**（守住白名单，不越权乱改）
- **不要求找到 N 个问题**（发现就登记，没发现不硬凑）
- **只要求发现了的不准遗失**（登记 + 闭环去向）
- 闭环去向可以是"延后""用户确认不管"——**不是所有问题都要立刻修**

它只堵住"遗失"这个洞，不扩张"审查"的范围。

### 2.6 拆解方法论（天玑核心职责）

> **反官僚主义**：鲁班的 EPIC→TASK→STORY→SPEC 四层太重。太一用两层（Task + 依赖序），核心约束是"一个 AI 会话能独立完成"。

#### 两层结构

```
Task（可执行单元）+ 依赖序（Task 间 DAG）
```

不要 EPIC/STORY/SPEC 多层分层。每个 Task 直接是天权/开阳的一个会话单元。

#### 拆解硬约束

每个 Task 必须满足：
1. **一个 AI 会话能独立完成**（天权/开阳的上下文边界）——这是判断"拆得对不对"的唯一标准
2. 有明确的 DONE（可机检验收）
3. 有明确的 DON'T（文件白名单 + 边界）
4. 标了 mode（文心/武毅）和风险标记
5. 标了依赖（哪些 Task 必须先完成）

天玑自问（判断 Task 大小）：
```
- 天权需要读多少文件才能开工？>10个 → 太大，拆
- 天权需要改多少文件？>5个 → 太大，拆
- DONE 能用一条 curl/一条测试命令验证吗？不能 → 太大或太模糊
- 和其他 Task 有循环依赖吗？有 → 重新切分边界
```

#### Task 按工作性质分三类（不是粒度分层）

| 类型 | 例子 | mode | 特点 |
|------|------|------|------|
| **构建型** | 新建登录模块、写 API、搭页面骨架 | tianquan | 从无到有，文心慢工 |
| **修复型** | 修 bug、性能优化、重构 | kaiyang | 从有到好，武毅速效 |
| **集成型** | 对接外部服务、数据迁移、联调 | 看情况 | 跨模块整合 |

#### 不同设计产出的拆法

**架构设计**：按模块/接口边界拆。每个独立模块或接口 = 一个 Task。
**UI 原型**：按页面/独立组件拆。每个可独立实现的界面单元 = 一个 Task。
**产品规格**：不直接拆——它先被天璇消化转化为 UI原型+架构设计，再由天玑拆。天玑拆的是天璇的最终产出，不是天璇的思考过程。

#### 依赖管理

Task 级 DAG（有向无环图）。司衡按拓扑序调度——无依赖的 Task 可并行，有依赖的串行。
```
Task-1(后端认证) ──┐
Task-2(后端密码) ──┼→ Task-4(前后端联调)
Task-3(前端表单) ──┘
```
不用 EPIC 层管理依赖。

### 2.7 三档授权（天玑核心职责 · 反官僚主义）

> **实证依据**：2025-2026 业界共识——spec 写"做什么+边界+验收"（just enough），不写"怎么做"（过度规格化是陷阱）。Anthropic 的"optimal altitude"、arXiv 警告"over-specification = pseudo-code"、Microsoft 数据"明确规格减少68%迭代"共同支持分层授权。详见搜索来源。

#### 核心原则：默认放权，只锁真正该锁的

太一的反官僚主义，体现在**默认倾向放权**。天玑拆解时的自问是"这个真的必须锁死吗？能放权吗？"，只有真正"错了 blast radius 大"的才锁死。

#### 三档授权（按 blast radius 分层）

| 档位 | 契约怎么写 | 天权自主权 | 适用 |
|------|----------|----------|------|
| 🔒 **锁死** | 行为描述（behavioral not procedural），精确到类型签名 | 0%，照做 | 接口契约/不变量/边界/验收 |
| 🔶 **约束** | 写约束 + 禁止项，边界模糊时打回天玑问 | 约束内自选，模糊则问 | 算法/库选择/模式 |
| 🔓 **放权** | 不写 | 完全自主 | 命名/内部结构/辅助函数 |

**锁死项判断标准**（错了 blast radius 大）：
- 接口契约（API 签名/数据格式）——错了影响所有调用方
- 不变量（业务规则/状态机）——错了破坏数据一致性
- 约束边界（DON'T/白名单）——错了越权乱改
- 验收标准（DONE）——错了不知道做完没

**放权项判断标准**（错了好改）：
- 实现模式（hooks/class/函数式）
- 命名/组织结构
- 内部辅助函数

#### 契约三段式结构（写进 contract-template.md）

```markdown
## Task: TASK-xxx

### 🔒 锁死项（天权 0% 自主权，必须照做）
- 接口：POST /api/login，body {email, password, mfaCode?}，返回 {token, expiresIn}
- 不变量：密码 bcrypt cost≥10；失败3次锁定15min
- DONE：curl /api/login 返回200+JWT；3次失败后第4次返回429
- DON'T：只改 src/auth/*；不改 user 表

### 🔶 约束项（天权在约束内自选，模糊则问）
- JWT 库自选，但必须支持 refresh
- 错误响应格式遵循现有 ApiError 类型

### 🔓 放权项（天权完全自主，契约不写）
- 内部函数组织、命名、是否抽 utils
```

#### 写法纪律（汲取三家竞品精华）

- **锁死项用行为描述，不用过程描述**（Matt Pocock 的 behavioral not procedural）
  - 好："提交时若 mfaRequired 为真，必须校验 mfaCode"
  - 坏："在 handleSubmit 里加个 if 判断"
- **锁死项禁文件路径/行号**（耐久契约纪律 11.3）
  - 好："LoginRequest 类型应增加 mfaCode 可选字段"
  - 坏："修改 src/auth/login.ts 第42行"
- **约束项加"Ask first"兜底**（agent-skills 的 Ask first 机制）
  - 天权遇到约束项边界模糊时，不擅自决定，打回天玑澄清
- **放权项尽量多**（反官僚主义的核心）
  - 低风险的全放，只锁真正高风险的

### 2.8 天枢方法论：四步判需求

> **融合三家精华**：SDD 的假设浮现 + 问道的三阶收敛 + idea-refine 的反谄媚/Not Doing + 天枢独创的正判。比任何一家都完整，"判"字落在正判环节，名副其实。

#### 核心流程

```
混沌 → 浮假设 → 炼痛点 → 正判 → 凝练真言
```

#### 第一步·浮假设（汲取 SDD · 防"默默填空"）

天枢收到混沌后，**先不追问，先列出自己脑补的假设**：

```
我听到的：{用户原话}
我假设的：
1. 这是给{谁}用的
2. {某个没说清的细节}我猜是{X}
3. 我假设{某个约束}成立
→ 这些假设对吗？哪个不对，现在纠正我。
```

**为什么这步最重要**：LLM 最危险的失败是"默默填空"——用户没说的它自己补了，还补错了。先浮假设让用户确认/纠正，从源头堵住。汲取 SDD 的 "Surface assumptions immediately / correct me now or I'll proceed"。

#### 第二步·炼痛点（汲取问道第一阶 · 抽象落到具体）

假设确认后，把抽象落到具体：

```
- 你遇到什么问题？说一个具体场景。
- 上次卡住是什么时候？你在做什么？
- 如果这个问题消失了，你的什么会变好？
```

完成标志：用户说出一句可以写进 Issue 的话。

#### 第三步·正判（天枢独创 · "判"的核心）

这一步问道没有，是天枢的增值。基于痛点做三个判断：

**判一·值不值得做**
```
- 如果不解决这个问题，会怎样？（检验真痛点 vs 伪痛点）
- 解决了的收益 vs 不解决的代价，值吗？
```

**判二·是不是伪需求（XY 问题）**
```
- 用户说想要 X，但真正需要的是 Y 吗？
- "你是想要 X，还是不想让 Y 发生？"（区分正向需求 vs 负向逃避）
```

**判三·边界在哪**
```
- 如果只解决一半，够不够？（找最小可行）
- 哪些看似相关但其实不该做？（Not Doing 清单）
```

完成标志：伪需求被显式斩掉（附理由），真需求被锁定，边界明确。

#### 第四步·凝练真言（汲取问道第三阶 · 压缩到最简）

```
- 如果只能用一句话描述你要什么？
- 这句话里哪个词可以删？
- 把这句话给一个新人，他能理解吗？
```

完成标志：需求可写成一句需求真言。

#### 需求真言产出格式（融合三家）

```markdown
## 需求真言

**混沌原念**: {用户最初说的那句话，原样保留}
**我假设的（已确认/已纠正）**: {第一步浮的假设，标注用户确认了哪些、纠正了哪些}
**所求之道**: {凝练后的一句话，用户真正要什么}
**筑基之痛**: {具体痛点场景，可写进 Issue}
**真需求**: {最小可行需求范围}
**伪需求（已斩）**: {正判中发现不需要的，显式排除 + 排除理由}
**Not Doing**: {看似相关但本次不做的 + 为什么不做}
**未明之境**: {还没想清楚的，留给下一轮}
**用户确认**: {用户是否确认此真言}
```

相比问道多了"假设浮现"和"Not Doing"，相比 idea-refine 多了"正判"和"伪需求显式斩"。

#### 反谄媚（汲取 idea-refine · 诚实判官）

天枢是**诚实的判官，不是 yes-machine**。如果需求不合理，温和但直接地指出：

```
❌ 谄媚："这个想法很好！"
✅ 诚实："我听到你说要做X，但从痛点看，你其实需要的是Y。做X是绕路，建议直接做Y。"
```

汲取 idea-refine 的 "Be honest, not supportive. A good ideation partner is not a yes-machine."

#### 降级信号（汲取问道 · 人机协作鲁棒性）

用户迷茫/烦躁/沉默时的应对：

| 信号 | 回应 |
|------|------|
| "我不知道" | 不知道很正常。先说说你最近一次觉得不对劲是什么时候？ |
| "别问了太多了" | 好，先停在这。我把目前摸到的整理一下，你看看有没有偏差。 |
| "就这样吧先做着" | 可以先行，但我标注了未明之境。如果后续方向偏了，随时回来再问一轮。 |
| 沉默 | 不急，想到什么说什么。 |

#### 各步可跳过（反官僚主义）

四步不是死流程——简单需求可跳步：
- 需求极明确（用户一句话就说清了）→ 跳过炼痛点，直接正判+凝练
- 假设很少（用户表达完整）→ 浮假设快速过
- **唯一不可跳的是正判**——无论需求多明确，都必须过一遍"值不值得做/是不是伪需求/边界在哪"

### 2.9 三星联合审查（玉衡·洞明·隐元）

> 玉衡/洞明/隐元都涉及"检查+驳回"，有共性骨架，但审查维度不同。本节统一梳理三者的分工、共性路由、以及与瑶光的串联。

#### 三星审查维度对比

| | 玉衡（自检）| 洞明（质门）| 隐元（风险守护）|
|---|---|---|---|
| **审谁** | 自己（天权审自己）| 天权（独立第三方）| 天权的代码（专项）|
| **审什么** | "我跑通了吗"——功能性自测 | "真的对了吗"——独立验证+设计评审+收尾 | "会暗中出事吗"——非功能性风险 |
| **标准** | 自测通过（低）| 独立证据（高）| 具体场景实证 |
| **能否放行** | 不能（不过不commit）| **能**（放行→push归档）| 不能（结果交洞明决策）|
| **触发** | 编码后必过（内化）| commit后必过+设计评审前置 | 按风险标记触发 |

**关键边界**：
- 玉衡是开发者自测，不是 QA（QA 是洞明）
- 隐元不直接放行——是洞明的专项扫描仪，结果交洞明综合决策
- 洞明是唯一有放行权的审查星

#### 三星串联关系

```
天权编码 → 玉衡自检（我跑通了吗）
  ↓ 自检通过才 commit
洞明审查（真的对了吗）
  ├─ 按风险标记触发隐元（专项风险扫描）
  │   └─ 隐元结果交回洞明（隐元不直接放行）
  ├─ 放行 → push/merge → 归档 → 收尾
  └─ 打回 → 瑶光查根因 → 路由
```

#### 玉衡分层自检（按 Task 类型，不全查）

| 检查项 | 所有Task | 构建型 | 修复型 | 集成型 |
|--------|---------|--------|--------|--------|
| 契约 DONE 逐条对照 | ✅ | ✅ | ✅ | ✅ |
| 构建成功（build exit 0）| ✅ | ✅ | ✅ | ✅ |
| 白名单内（没越界）| ✅ | ✅ | ✅ | ✅ |
| 单元测试通过 | — | ✅新代码 | ✅修复点 | 看情况 |
| E2E/冒烟（真实触发）| — | ✅主流程 | ✅复现原bug | ✅集成路径 |
| 前端视觉校验 | — | ✅如有UI | — | ✅如有UI |

**核心原则**（汲取 superpowers verification-before-completion）：
- **贴证据不贴声称**——"测试通过"必须附 `[Run xxx] [See: 34/34 pass]`，不接受"should pass"
- **跑过的才算**——本次会话没跑过验证命令，不能声称通过
- **新鲜证据**——不引用上次会话的输出

**前端校验分层**（前端无法用 curl，但可分层）：
| 层 | 怎么验证 | 证据 |
|----|---------|------|
| 组件逻辑 | 单元测试（jest/vitest）| 测试输出 |
| 交互行为 | 组件测试（testing-library）| 测试输出 |
| 视觉呈现 | 启动后截图 / DOM 断言 | 截图或断言输出 |
| 真实流程 | E2E（Playwright/Cypress）| 测试输出 |
| 兜底 | 手动操作描述 + 预期结果 | 操作步骤+结果 |

诚实边界：无自动化测试设施时，玉衡至少描述手动验证步骤和预期，不能说"界面应该没问题"。

#### 共性驳回路由（三星共用，按根因不按发现者）

| 根因 | 驳回给 | 玉衡处理 | 洞明处理 | 隐元处理 |
|------|--------|---------|---------|---------|
| **实现问题**（代码bug）| 天权 | 自修（≤2轮）| 打回天权 | 交洞明→打回天权 |
| **契约歧义**（DONE模糊）| 天玑 | 打回天玑补清 | 打回天玑 | 交洞明→打回天玑 |
| **架构偏差**（偏离技术方案）| 天璇 | 打回天璇 | 打回天璇 | 交洞明→打回天璇 |
| **契约外影响**（越界）| 登记 | 登记intervention | 登记intervention | 登记intervention |
| **需求问题**（需求本身有误）| 天枢 | — | 升级用户→天枢 | — |

**为什么按根因不按发现者**：同一个架构偏差，无论玉衡发现还是洞明发现，都该回天璇修。路由标准是"问题出在哪颗星"，不是"谁发现的"。

#### 与瑶光的串联

```
玉衡/洞明/隐元 发现问题 → 按根因路由（上表）
  ├─ 实现问题 → 天权修 → 玉衡自检 → 重新交洞明
  ├─ 契约/架构/需求问题 → 回对应星修 → 重走流程
  └─ 洞明打回且根因复杂 → 瑶光查根因 → 路由回源头星
```

玉衡是自检层的第一道路由，洞明是独立审查层的第二道路由，瑶光是打回后的根因深挖。三层递进，各管一段。

### 2.10 瑶光方法论：双面诊断

> **融合三家精华**：玄鉴的双面（bug+架构）+ superpowers 的"3次修复失败质疑架构" + Matt Pocock 的"反馈回路为核心"。比任何一家都完整。

#### 面一·照骨（bug 根因诊断）

```
摄迹（建反馈回路）→ 寻脉（定位坏味）→ 探源（因果链定根因）
```

**第一步·摄迹——构建反馈回路（汲取 Matt Pocock Phase 1，这是核心）**

瑶光收到 bug 报告后，**先不读代码猜原因，先构建一个可复现的 pass/fail 信号**。这是整个诊断的核心——有快速确定性的反馈回路，bug 就能找到；没有，盯代码再久也没用。

按顺序尝试构建反馈回路：
1. 失败测试（unit/integration/e2e）
2. curl/HTTP 脚本打 dev server
3. CLI 调用 + fixture 输入
4. headless 浏览器脚本（前端 bug）
5. 重放捕获的 trace
6. 一次性 harness（最小子集 + mock）
7. fuzz/属性测试（间歇性 bug）
8. bisection（两个已知状态间二分）

目标：一个快速、确定性、可复现的 pass/fail 信号。**没有这个信号，不进入下一步——先建回路。**

**瑶光铁律**（汲取 superpowers）：
- NO FIXES WITHOUT ROOT CAUSE（无根因不修复）
- 没有反馈回路不猜测根因
- 错误信息要逐字读（常含解法）、堆栈要完整读

**第二步·寻脉——定位代码坏味**

有了反馈回路后，循证据定位可疑代码，堪破代码坏味（汲取玄鉴五衰，已改名中性）：
| 坏味 | 表现 |
|------|------|
| 过度特化 | 只对某个 case 生效，换输入就崩 |
| 巨石回归 | 函数/类过大，改一处动全身 |
| 虚假测试 | 测试永远 pass，没覆盖真路径 |
| 教条实现 | 照搬模式不顾实际上下文 |
| bug 复现 | 修了又犯，同类问题反复 |

**第三步·探源——因果链定根因**

构建因果图，沿数据流反向追溯（汲取 superpowers root-cause-tracing）：
```
- 错误值从哪产生？
- 谁用错误值调用了这里？
- 一路追到源头
- 在源头修，不在症状修
```

**3次失败质疑规则**（汲取 superpowers 最有价值的一条）：
```
瑶光定位根因后，若修复 3 次仍失败：
  → 不是假设错了，是架构错了
  → 停止修复，质疑架构
  → 路由到天璇重新决断（或登记 intervention 待架构重构）
  → 与用户讨论，不独自磨第4次
```

这条规则直接回答"修不好的 bug 交给谁"——**修不好的 bug 根因在架构，交给天璇**。

#### 面二·观势（架构体检）

```
观势（模块边界/依赖图）→ 审阵（六合打分）→ 寻脉（复杂度热点）
```

**观势**：照出模块边界、依赖图，显形巨石/上帝类。
**审阵**：六维打分——伸（可扩展性）/御（安全性）/续（可维护性）/疾（性能）/阵（部署）/牒（文档）。
**寻脉**：定位复杂度热点，堪破代码坏味。

主要用于 **Epic 完成后的架构回归**——全局审视接口干净度、实现偏离度。偏离→回天璇重新评估。

#### 根因精准路由（瑶光的核心价值）

瑶光诊断完不直接修复（瑶光不改代码），而是**按根因精准路由**：

| 根因定位 | 交给谁 | 场景 |
|---------|--------|------|
| **实现偏差** | 天权/开阳（直接修）| 代码 bug，按契约修即可 |
| **架构问题** | 天璇（重新决断）| 3次修不好/接口设计有问题/模块耦合 |
| **契约歧义** | 天玑（补清契约）| DONE 模糊导致实现跑偏 |
| **需求问题** | 天枢（重新正判）| 需求本身就错了（伪需求没斩干净）|
| **非功能性风险** | 隐元（专项深挖）| 安全/并发/性能隐患需专项审计 |

**为什么精准路由而非无差别甩给某颗星**：同一个"登录失败"，可能是代码 bug（→天权）、可能是架构选错（→天璇）、可能是需求理解错（→天枢）。瑶光的价值正在于**区分根因并路由到正确的星**，不笼统打回。

与 2.9 节共性驳回路由表一致——路由按根因不按发现者。

#### 两个触发场景

**场景一·被动触发（洞明打回）**：
```
洞明打回 → 瑶光查根因 → 路由建议经司衡执行 → 回源头星修复
```
流水线内——洞明审查发现实现不对但不确定根因，交瑶光深挖。路由建议经司衡执行（自动续接）。

**场景二·主动召唤（用户 @瑶光）**：
```
用户 @瑶光 这个报错怎么回事
  → 瑶光照骨（建回路→定位→定根因）
  → 产出根因报告 + 路由建议
  → 报告交用户决策（A 路径，不自动续接）
```
用户直接 @ 瑶光诊断，诊断完不自动续接。用户决定下一步（交给谁修，或 @司衡 继续）。

**两场景的区别**：被动触发路由经司衡执行；主动召唤只出报告，用户决定。

#### 诊断报告格式

```markdown
## 瑶光显影（诊断报告）

**症状**: {用户/洞明报告的问题}
**反馈回路**: {如何复现——测试命令/curl/脚本，pass/fail 信号}
**坏味定性**: {过度特化/巨石回归/虚假测试/教条实现/bug复现}
**根因**: {沿因果链追溯到的源头，精确到哪颗星的问题}
**凶险定级**: P0紧急 / P1重要 / P2建议
**路由建议**: {交哪颗星修复 + 理由}
**修复后验证**: {如何确认根因已除——回照反馈回路}
```

### 2.11 天璇方法论：四步流程 + 三项设计

> **融合三家精华**：brainstorming 的探索上下文/多方案/分段呈现/自审/Design for isolation/YAGNI + SDD 的技术假设浮现/成功标准转化 + 司南的依赖盘点意识/约束可机检。天璇肩负产品+UI+架构三项设计，差之毫厘谬以千里。

#### 核心流程：探上下文 → 浮假设 → 探方案 → 定设计

三项设计（产品/UI/架构）不是三个独立流程，是**同一个设计流程的三个产出面**。

**第一步·探上下文（汲取 brainstorming · 不凭空设计）**

天璇收到需求真言后，**先不设计，先探索现有项目**：
```
- 读现有代码结构（模块/依赖/模式）
- 读近期 commit（了解在做什么）
- 读已有 ADR / 文档（了解既有决策）
- 识别既有约束（技术栈/框架/规范）
```
**为什么这步最重要**：凭空设计会和现有代码冲突。先理解现状，设计才能"长在"代码库里。

**第二步·浮技术假设（汲取 SDD · 技术层面的假设浮现）**

天枢已浮现了需求假设，天璇再浮现**技术假设**：
```
基于需求真言和项目现状，我假设：
1. 数据存储用{X}（基于现有schema）
2. 认证机制是{Y}（基于现有中间件）
3. 前端框架是{Z}（基于现有依赖）
→ 这些技术假设对吗？纠正我。
```

**第三步·探方案（汲取 brainstorming · 提2-3方案带trade-offs）**

天璇**不直接给单一方案，先给2-3个可行方向**，各带代价和推荐：
```
方案A：{描述} — 优点/缺点/适用场景
方案B：{描述} — 优点/缺点/适用场景
方案C：{描述} — 优点/缺点/适用场景
我推荐{X}，因为{理由}。但如果你更看重{Y}，方案{Z}更合适。
```
**为什么必须多方案**：单一方案让用户没得选（被动接受），多方案让用户参与决策（主动选择）。也防止天璇过早锁定可能错误的方案。

**第四步·定设计（汲取 brainstorming 分段呈现 + Design for isolation）**

方案选定后，**分段呈现设计，每段获确认再继续**：
```
段1：架构（模块划分/接口定义/数据流）→ 用户确认 → 继续
段2：产品（功能列表/用户旅程）→ 用户确认 → 继续
段3：UI（关键界面/交互说明）→ 用户确认 → 继续
（按需呈现，不是每项都做）
```

#### 三项设计的各自方法论

**架构设计**（怎么实现）：
```
- 模块划分：每个模块单一职责，接口清晰（Design for isolation）
- 接口定义：签名 + 数据格式 + 错误模式（不是只写函数名）
- 技术选型：必须有 ADR（记录决策+为什么+淘汰了什么）
- 约束清单：不可变接口 + 禁止项 + 不变量（给天玑做锁死项的依据）
- 数据流：请求怎么走、状态怎么变、错误怎么传
- 不做完美主义：权衡利弊做决策，不为最优无限拖延
```

**产品设计**（做什么功能）：
```
- 用户旅程：从用户视角走一遍完整流程（入口→操作→结果）
- 功能列表：必须做的 vs 可选的（YAGNI ruthlessly 砍可选）
- 验收标准草案：每个功能"做到什么样算完成"（给天玑 DONE 的雏形）
- 边界场景：正常流 + 至少3个异常流
```

**UI 原型**（用户看到什么）：
```
- 关键界面：核心页面的线框/布局描述（不需要高保真）
- 交互说明：点击什么→发生什么→跳转哪里
- 状态呈现：加载/成功/失败/空数据 各显示什么
- 与架构的接口：前端调哪些接口、传什么、收什么
```

#### 产出：设计包

```markdown
## 设计包

### 技术假设（已确认/已纠正）
{第二步浮的技术假设}

### 选定方案 + ADR
{第三步选定的方案 + 为什么选它 + 淘汰了什么}

### 架构设计
- 模块划分：{每个模块职责+接口}
- 接口定义：{签名+数据格式+错误模式}
- 约束清单：{不可变接口/禁止项/不变量}
- 数据流：{请求/状态/错误的流转}

### 产品设计（如需）
- 用户旅程：{入口→操作→结果}
- 功能列表：{必须做 vs YAGNI砍掉}
- 验收标准草案：{每个功能的完成定义}
- 边界场景：{正常流 + ≥3异常流}

### UI原型（如有UI）
- 关键界面：{线框/布局}
- 交互说明：{点击→发生→跳转}
- 状态呈现：{加载/成功/失败/空}
- 前后端接口：{调什么/传什么/收什么}

### 风险清单
{识别的风险 + 应对策略}
```

#### 自审（汲取 brainstorming Spec self-review）

设计包写完后，天璇自审四查：
```
1. 占位符扫描：有 TBD/TODO/模糊描述吗？修掉
2. 内部一致性：架构和产品描述矛盾吗？接口和数据流对得上吗？
3. 范围检查：太大需拆分吗？还是太小不够一个Task？
4. 歧义检查：任何需求能被两种方式理解吗？选一种明确写
```

#### 关键纪律

**纪律一·不拆Task**：天璇只设计，拆 Task 归天玑。天璇的产出是设计包，不是 Task 列表。

**纪律二·不做完美主义**：权衡利弊做决策，不为最优方案无限拖延。有 2-3 个够用的方案就选一个推进。

**纪律三·约束要可机检**：架构决策必须落地为可机检约束（不能只说"用JWT"，要写"认证返回 JWT，过期2h，字段token"）。这是给天玑做锁死项的依据。

**纪律四·探索现有代码先行**：不凭空设计，先理解现状。设计要"长在"代码库里，不和现有模式冲突。

**纪律五·YAGNI ruthlessly**：砍不必要的功能。每个功能都要问"真的需要吗？不做的代价是什么？"（汲取 idea-refine 的 Not Doing 精神）。

### 2.12 天权/开阳方法论：编码五步 + 双模式风格

> **融合三家精华**：鲁班的开工5分钟盘点/外科手术/异常熔断 + superpowers的四态报告/In over your head/self-review + incremental-implementation的Rule 0 Simplicity First/Rule 0.5 Scope discipline/增量循环/三种切片。天权/开阳是唯一写代码的角色，执行得对不对直接决定体系产出质量。

#### 编码五步（双模式共用）

```
开工盘点 → 增量实现 → 自验证 → 玉衡自检 → 报告
```

**第一步·开工盘点（汲取鲁班5分钟盘点 · 编码前必做）**

收到契约后、写代码前，强制自答5问：
```
1. 本次改动的代码，被谁调用？数据流上下游是什么？
2. 数据流上有没有"看起来正常但可能失效"的环节？（mock/fallback/默认值）
3. 此功能跑起来需要哪些外部依赖？每个本地能起吗？
4. 如果现在启动应用触发此功能，最大可能挂在哪一步？
5. 契约 DONE 里有没有"启动应用真实触发一遍"这一条？没有→自己加
```
**5问任一暴露"可能跑不通"→ 停手，写隐患报告交天玑/天璇，不开工。**

**第二步·增量实现（汲取 incremental-implementation · 不大块写）**

不大块写完才测，按薄切片增量推进：实现最小完整片 → 测 → 验证 → commit → 下一片。

切片策略（按 Task 性质选）：
- **vertical slice**（首选）：一个端到端可演示的路径
- **contract-first**（前后端并行）：先定接口契约，后端实现+前端mock，再联调
- **risk-first**（高风险 Task）：先做最不确定的部分，早失败早知道

**第三步·自验证（汲取 incremental Rule 1-2 + verification-before-completion）**

每片增量后：
```
- 构建成功（build exit 0）
- 现有测试不破（回归）
- 新功能可验证（测试/curl/手动）
- 贴证据不贴声称（本次跑过的才算）
```
**铁律**：每片增量后项目必须可编译、现有测试必须通过。不留破损状态。

**第四步·玉衡自检（内化 · 已在2.9节详述）**

按玉衡分层自检表（Task类型×检查项）逐项检查，通过后才 commit。

**第五步·报告（汲取 superpowers 四态 · 精确表达状态）**

```
DONE：全部完成，自检通过，附证据
DONE_WITH_CONCERNS：完成但有疑虑（附疑虑内容）
BLOCKED：无法完成（附卡在哪+已尝试什么+需要什么帮助）
NEEDS_CONTEXT：缺信息（附缺什么）
```
**绝不**：silent produce unsure work（不确定的代码不静默交付）。

#### 双模式风格分化

流程五步一样，但编码风格按模式分化：

| | 天权（文心·mode=tianquan）| 开阳（武毅·mode=kaiyang）|
|---|---|---|
| **场景** | 新建模块、核心逻辑、架构实现 | 修bug、性能优化、攻坚难题 |
| **风格重心** | 重"理"——工整、可读、可维护 | 重"效"——直击要害、最小改动 |
| **命名** | 精确反映"做什么"，不反映"怎么做" | 清晰即可，不纠结完美命名 |
| **抽象** | 三次重复才抽象（Rule 0）| 能不改结构就不改，局部修复 |
| **测试** | TDD（红绿重构）| 先复现bug（红）→修→验证（绿）|
| **提交粒度** | 按逻辑单元提交 | 按修复点提交，最小diff |
| **速度取向** | 慢工出细活 | 兵贵神速 |

#### 关键纪律

**纪律一·Simplicity First（汲取 incremental Rule 0）**
```
写代码前问：最简可用的方案是什么？
写完后问：能不能更少行？这个抽象配得上它的复杂度吗？
✗ 为一个通知造通用EventBus+中间件管线
✓ 简单函数调用
```
三行重复好过过早抽象。先写朴素正确的版本，证明对了再优化。

**纪律二·Scope discipline（汲取 incremental Rule 0.5）**
```
只碰任务要求的。发现了任务外的问题→记录不修：
"NOTICED BUT NOT TOUCHING:
 - src/utils/format.ts 有未用import（非本任务）
 - auth中间件错误信息可以更好（单独Task）
 → 要我建Task吗？"
```
登记 intervention.md（走溢出机制），不擅自修。

**纪律三·外科手术式修改（汲取鲁班）**
只改契约白名单内的文件。白名单外的改动=越权。

**纪律四·熔断（汲取鲁班）**
```
3次不同方案仍无法达成→停，输出《阻塞报告》：
- 试了什么方案
- 每次失败在哪
- 我的判断（契约问题/架构问题/能力不足）
交司衡→路由给天玑/天璇/用户
```
不空转第4次。

**纪律五·In over your head（汲取 superpowers）**
```
该停手求助的信号：
- 任务需要架构决策（多个有效方案）→ 不是天权/开阳的活，回天璇
- 需要理解提供的上下文之外的代码 → NEEDS_CONTEXT
- 不确定自己的方案对不对 → BLOCKED
- 连续读很多文件还没搞懂系统 → BLOCKED
```
坏代码比没代码更糟。求助不受惩罚。

#### commit 规范

```
<type>(<scope>): <TASK-ID> — <摘要> + <证据锚点>

type: feat/fix/refactor/test/docs/chore
scope: 模块名
TASK-ID: 契约的Task标识
摘要: 一句话
证据锚点: 测试命令+结果（如 pytest -k login → 5/5 pass）
```

## 三、司衡：调度中枢（斗为帝车）

### 3.1 定位与边界

司衡是"斗为帝车，运于中央"的调度中枢。本质是**优先级队列调度器**（类比单核 CPU 调度器）——读请求、判断派谁、推进流程、管理工作区。

**司衡做的**（路由+调度+工作区管理）：

| 动作 | 说明 |
|------|------|
| 语义匹配路由 | 读九星能力卡片，把请求语义匹配到合适的星 |
| 状态感知调度 | 读 status/task-queue，按五指规则推进流水线 |
| 优先级队列管理 | 三档优先级（urgent/normal/low），urgent 抢占 low |
| 工作区管理 | 抢占时管理 WIP 分支/worktree，恢复时 rebase |
| 前置检查 | 路由前验证目标星可接收（如天权需契约）|
| 断点续接 | 读已有文档判断进度，从断点续接 |
| 元问题直接答 | 状态/队列/遗留问题（读文件直接答）|
| 流程控制 | 停/改派 |

**司衡拒绝做的**（所有执行性工作）：

| 动作 | 归属 | 司衡动作 |
|------|------|---------|
| 写代码 | 天权/开阳 | 路由天权（但需契约，缺则告知）|
| 审查代码 | 洞明 | 路由洞明 |
| 诊断bug | 瑶光 | 路由瑶光 |
| 设计方案 | 天璇 | 路由天璇 |
| 拆解任务 | 天玑 | 路由天玑（流水线内调度）|
| 判需求 | 天枢 | 路由天枢 |
| 代码解读/技术问答 | 建议直接问或走瑶光 | 承认非职责 |

司衡**不常驻**——只在被调用时运行。**无状态**——每次从文件重读，不依赖对话记忆（扛 LLM 上下文压缩）。

### 3.2 智能路由机制（三层咬合）

**第一层·意图分类（语义匹配）**

司衡 SKILL 里嵌入九星能力卡片，LLM 读到请求后做语义匹配：
```
天枢：判需求——用户表达想做某功能/想要某能力/觉得缺什么/需求模糊
天璇：设计——用户要技术方案/架构讨论/UI设计/产品设计
洞明：审查——用户要审查代码/审PR/清理工作区/归档收尾
瑶光：诊断——用户报告bug/问怎么回事/要查根因/架构评估/代码解读
（天玑/天权/玉衡/隐元仅内部调度，用户不直接触发）
```
**司衡不是无脑把请求丢给天枢**——它读九星定义做语义匹配，路由到最合适的星。

**第二层·状态感知（读 status/task-queue）**

语义匹配 + 当前状态综合判断：
- 请求含明确意图（"做功能""查bug"）→ 语义匹配为主
- 请求是流水线推进（"继续""下一步"）→ 读 status 按五指规则
- 请求是"完成已有需求"→ 读已有文档判断断点

**第三层·前置检查（路由前验证目标星可接收）**

```
路由天权 → 天权需契约 → task-queue 有 ready Task？
  ├─ 有 → 路由天权
  └─ 没有 → 告知"天权需契约，请先完成天枢→天璇→天玑"
```

### 3.3 路由决策矩阵

| 请求类型 | 第一层匹配 | 第二层状态 | 路由到 |
|---------|----------|----------|--------|
| 新需求（"做功能"）| 天枢 | — | 天枢（天枢自己分层：明确走轻量/模糊走完整）|
| 继续/下一步 | — | 读status | 按五指规则 |
| bug/报障 | 瑶光 | — | 瑶光 |
| 审查/审PR | 洞明 | — | 洞明 |
| 架构讨论 | 天璇 | — | 天璇 |
| 完成已有需求 | — | 读已有文档判断断点 | 断点的下一颗星 |
| 当前状态 | 元问题 | — | 司衡直接答 |
| 遗留问题 | 元问题 | — | 司衡直接答 |
| 停/改派 | 控制指令 | — | 执行控制 |
| 代码问题 | 九星都不匹配 | — | 承认非职责，建议 |
| 写代码 | 天权 | 需契约 | 检查契约→有则路由/无则告知缺前置 |

### 3.4 五指调度规则

| 规则 | 触发 | 动作 |
|------|------|------|
| **指极** | 用户提需求 | 先清点 intervention.md 历史 open 条目（摆给用户决策）→ 路由天枢起步 |
| **指营** | 天玑产出契约、有 ready Task | 调度天权/开阳营建 |
| **指校** | 营建玉衡自检通过、commit | 调度洞明质门审查 |
| **指通** | 洞明放行+push+归档完成 | 取队列下一个 Task / 标记 Epic done |
| **指回** | 洞明打回 | 调度瑶光查根因 → 按根因路由回源头星（最多2轮，第3轮升级用户）|

### 3.5 自动调度机制

`@司衡` 启动后，**除澄清问题外全程自动调度**。

**自动续接的唯一机制：星完成时显式调用司衡。**
```
司衡调度某星 → 星执行 → 星完成
  → 星最后一步：更新 status + 调用 Skill("siheng")
  → 司衡读 status → 按五指规则派下一颗
  → 循环，直到停止点
```
每颗星 SKILL 末尾必须写："完成产出文档后，更新 status 并调用司衡"。没有这步，自动调度断链。

**无轮询/定时检查**——SKILL 体系无进程调度权，不假装有兜底。

**三种停止点**：澄清问题 / 熔断（3次失败或打回）/ 流程完成。

**用户干预权**：`@司衡 停` / `@司衡 改派XX`。

### 3.6 优先级队列（单核调度器模型）

task-queue.md 是带优先级的队列，status.md 记当前运行态。类比单核 CPU 调度器——有优先级、有抢占、有上下文切换，但**串行不并发**（一次只执行一个 Task）。

**三档优先级**：

| 优先级 | 场景 | 抢占权 |
|--------|------|--------|
| **urgent** | 生产bug/数据损坏/安全漏洞 | 可抢占任何低优先级 |
| **normal** | 功能开发Task（天玑拆的默认值）| 不抢占，按队列顺序 |
| **low** | 优化/重构/技术债（intervention升级的）| 只在队列空时执行 |

**谁定优先级**：天玑拆 Task 时标 normal；用户/洞明可升 urgent（紧急bug）；intervention 升级的延后 Task 默认 low。

### 3.7 抢占与工作区管理

代码 Task 共享同一个工作目录——抢占/并行不只是切上下文，还要处理**文件级隔离**。

**抢占式（串行，紧急 bug 插入功能开发）**：

```
天权在 TASK-001 编码（normal）→ 紧急 BUG-001 来了（urgent）
  → 抢占：
    1. 天权把半成品 commit 到 wip/{TASK-ID} 分支（不 push，耐久可靠）
    2. checkout 回主分支
    3. TASK-001 移入挂起区（记录 wip 分支+恢复动作）
    4. status 切到 BUG-001
  → 瑶光诊断 + 天权修 BUG-001（主分支或 hotfix 分支）
  → BUG-001 完成 merge 回主分支
  → 恢复 TASK-001：
    1. checkout wip/{TASK-ID}
    2. rebase 主分支（拿最新代码，含 hotfix 改动）
    3. 解 rebase 冲突（如有）→ 天权续接
```

**推荐 WIP 分支而非 stash**——stash 易丢失，commit 到 WIP 分支更耐久可靠（呼应耐久契约原则）。

**并行式（选配，需平台支持多会话）**：

```
无依赖的多个 Task → git worktree 各开独立工作区
  worktree-1: TASK-001 在 .worktrees/task-001/
  worktree-2: TASK-002 在 .worktrees/task-002/
多个天权会话并行 → 各自完成后 merge → 冲突交天权解
无平台支持 → 退化为串行排队
```

**三条工作区规则**：
1. **正常串行**（默认）：一个 Task 做完 commit → 下一个同分支继续，无需 stash/worktree
2. **抢占**：commit 到 wip/{TASK-ID} 分支 → checkout → 恢复时 rebase
3. **并行**（选配）：git worktree 独立工作区，需平台支持

### 3.8 状态文件

```
读：status.md、task-queue.md、intervention.md、epics/（断点续接时）
写：status.md、task-queue.md
不写：各星产出文档（由各星自己写）
```

### 3.9 僵尸检测（被动检查，非看门狗）

**不设看门狗**——SKILL 体系无进程调度权，同进程定时器会被阻塞。不假装能提供做不到的能力。

改用被动检查——司衡在被调用时顺便检查 status 的 last_update，若 >阈值无更新报告"疑似僵尸"交用户决定。

**诚实边界**：被动检查不承诺"及时发现僵尸"，只承诺"被调用时会检查"。需外部调度器（os cron/CI/PM2，用户自部署）才能补足。

## 四、文档链：九星之间的接力

### 4.1 核心文档集（从下游倒推，无孤岛）

```
需求真言 ← 技术方案 ← 任务契约 ← 代码+自检 ← 测试报告 ← 放行单
  天枢      天璇        天玑      天权/开阳+玉衡   洞明       洞明
                                                (+隐元非功能性风险报告)
```

每个文档是下一颗星的**硬门禁输入**——读不到就拒绝执行。

### 4.2 LLM 友好的文档四原则

| 原则 | 为什么 | 怎么做 |
|------|--------|--------|
| **结构化** | LLM 解析表格/列表远好于自然语言段落 | Markdown 表格 + YAML frontmatter + 固定章节 |
| **机检字段** | LLM 会"声称符合"，必须能 grep/test 验证 | DONE 写可勾选 `[ ]`、文件白名单写绝对路径 |
| **显式引用** | LLM 不会自己建立文档间关联 | `upstream: [需求真言.md]` `downstream: [营建]` |
| **状态自描述** | LLM 读文档要立刻知道该不该处理它 | 头部 `status: draft/done/approved` + `next_action` |

## 五、对 LLM 友好的三个保证

### 保证一：上下文隔离

每颗星 = 独立 SKILL + 独立会话边界。只加载自己的 SKILL + 自己的输入文档，不背全流程上下文。

### 保证二：文档接力

星与星之间靠文档交接，不靠对话记忆。即使跨会话、被压缩、重启，状态全在文件里。

### 保证三：门禁可机检

每颗星的门禁是 grep/test 级别的机械检查，不靠 LLM "自觉"。

## 七、入口与命令

### 7.1 双入口机制（司衡 + 开放星宿）

体系有**两类入口**，语义明确、互不冲突：

| 入口 | 语义 | 是否续接 | 场景 |
|------|------|---------|------|
| `@司衡` | "启动帝车，帮我跑流程" | **自动续接**（帝车运于中央）| 流水线作业（做功能/执行/审查/归档）|
| `@星宿` | "我只要这一步" | **不续接**（星完成即停）| 交互性场景（澄清需求/debug/讨论方案）|

**零歧义**：用户用哪个入口，就表达了是否要续接。不需要司衡猜，不需要用户额外声明。

### 7.2 司衡入口（流水线模式）

`@司衡` 启动帝车，**语义路由 + 自动续接**：

```
@司衡 <任意请求>
  ↓
司衡读 SKILL 内的九星定义 → 请求与九星语义匹配
  ├─ 强匹配某星 → 直接路由（如"bug"→瑶光）
  ├─ 匹配"流水线推进" → 读status派下一颗（如"继续"）
  ├─ 多星弱匹配 → 问用户（"先需求还是先审查？"）
  └─ 无匹配（纯问答）→ 司衡直接答或问用户
  ↓
路由到星 → 星执行 → 星的最后一步：更新status + 调用Skill("siheng")
  ↓
司衡被调用 → 读status → 按五指规则自动派下一颗（用户不用手动接力）
  ↓
持续运转，直到：澄清问题 / 熔断 / 流程完成
```

**关键**：司衡是语义路由器——它读九星定义做意图分类（轻量匹配，不是深度需求理解），把请求路由到最合适的星。意图理解是被路由到的星的事。

**路由容错**：星收到后发现自己不该处理 → 打回司衡重新路由。

**用户主动干预权**（随时可用）：
```
@司衡 停              → 中止当前自动调度
@司衡 改派天璇         → 强制路由到指定星
```

### 7.3 星宿入口（单星直达模式）

部分交互性强的星对用户开放，**直达不续接**：

| 开放的星 | 触发 | 场景 |
|---------|------|------|
| `@天枢` | `/taiyi:tianshu` | 需求澄清——聊到一半发现问题就不执行了 |
| `@天璇` | `/taiyi:tianxuan` | 架构讨论——只想讨论方案，不下定论 |
| `@洞明` | `/taiyi:dongming` | 独立审查 PR / 独立清理工作区（归档/收尾/状态重置）|
| `@瑶光` | `/taiyi:yaoguang` | debug 诊断——只查根因，看完报告自己判断要不要修 |

**不开放的星**（只能由司衡调度）：
- 天玑（拆任务需明确输入）、天权/开阳（编码需契约输入）、玉衡（内部自检）、隐元（非功能性风险守护子流程）

这些是流水线节点，用户直接调无意义（没有契约你调天权写啥代码？）。

### 7.3.1 入口分层原则（所有开放入口共用）

> **修正"入口=固定场景"的设计瑕疵**：每个开放入口都能轻量处理，不强制走完整流程。按请求性质决定深度，能做就做，超出职责才建议走别的入口。

| 入口 | 轻量（直接做）| 完整（按需）| 超出职责 |
|------|-------------|-----------|---------|
| **天枢** | 只正判（这需求合不合理）| 四步完整（混沌→真言）| 技术方案（→天璇）|
| **天璇** | 明确需求轻量正判+设计 | 模糊建议→天枢 | 审已交付代码（→洞明）|
| **洞明** | 无契约按通用标准审 | 有契约对照契约审 | 改代码（→天权）/debug（→瑶光）|
| **瑶光** | 解读（为什么这么写）/评估（有没有隐患）| 诊断（有症状bug）/观势（架构体检）| 改代码（→天权）|
| **司衡** | 元问题（状态/队列/遗留）| 流水线自动调度 | 领域深问（建议走对应星）|

**天璇轻量正判的触发条件**：
- 需求来自天枢（有需求真言/摘要）→ **跳过正判**，直接进设计（天枢已做过）
- 需求来自用户直达（@天璇，未经天枢）→ **轻量正判**再进设计
- 正判永远只做一次——要么天枢做（@司衡路径），要么天璇做（@天璇直达路径）

**司衡路由天枢后，天枢自己分层**：
- 明确需求（"注册功能，手机号+邮箱"）→ 轻量正判（快速检查有没有坑）
- 模糊需求（"用户系统不太好"）→ 完整四步

### 7.4 续接：A 路径

用户经 `@星宿` 单星完成后，若想继续往下走，**手动召唤司衡接管**：

```
@天枢 帮我澄清需求
  → 天枢和用户对话澄清 → 产出需求真言 → 结束（不续接）

用户：@司衡 继续（带着需求真言）
  → 司衡接管 → 调度天璇 → ... 自动流转
```

**A 路径的设计意图**：单星模式默认不续接（尊重用户"只做这一步"的意图）；用户要续接时显式召唤司衡，**模式切换由用户主动触发，不由星或司衡替用户决定**。

### 7.5 太一契约文档（非入口）

太一是契约文档，不是可召唤的入口。它在体系中的存在形式：

```
docs/taiyi-school/contracts/
  └── 太一.md（或按业务族分文件）
      ---
      status: sacred（已确立即神圣不可擅改）
      ---
      ## 契约内容（TASK-ID + DONE + DON'T + 扩展字段）
```

- 由天玑（拆任务）确立，确立后神圣，擅改须经司衡回到天枢/天璇重新对准
- 九星各自读取太一契约的相关部分作为工作依据
- 司衡每次调度前参照太一契约，确保星工作不偏离

### 7.6 设计评审

**不独立成星，洞明前置承担**。洞明 SKILL 分两模式：
- `规划域完成时`：读需求真言+技术方案+契约，审设计合理性 → 放行进营建 / 打回回天枢天璇
- `营建commit时`：读代码+契约，审行为对不对 → 放行进归档 / 打回进瑶光

两模式共享"独立审查"方法论，只是输入不同。保持九星完整，不增设第十颗星。

### 11.1 洞明审查升级：doubt-driven 对抗复审

**来源**：agent-skills 的 doubt-driven-development（整个仓库的皇冠机制）

**解决的问题**：洞明"独立审查"目前只写了"禁止信任天权单方声称"，但没写**怎么保证独立**——如果洞明读了天权的自检报告，它会不自觉地被报告的结论带偏（确认偏误）。

**落地的五步循环**（写进洞明 SKILL）：

```
洞明审查一个非平凡交付时，走 doubt 循环：

Step 1·CLAIM（天权命名决策）
  天权的自检报告里声明："本次改动线程安全/符合契约/边界覆盖"
  → 洞明记录这个 CLAIM，但不采信

Step 2·EXTRACT（剥离推理，只取事实）
  洞明只提取：代码 diff（artifact）+ 任务契约（contract）
  ★ 关键：不传天权的自检报告/CLAIM/推理过程
  传结论只会得到对结论的验证，不传结论才能独立判断

Step 3·DOUBT（对抗式审查）
  洞明用对抗提示词审查 artifact+contract：
  "找出这个代码的问题。假设作者过度自信。找：
   - 未声明的假设
   - 未处理的边界
   - 隐藏的耦合/共享状态
   - 契约可能被违反的方式
   - 意外输入下的失败模式
   不要验证。不要总结。找问题，或明确说找不到。"
  ★ 对抗提示词覆盖洞明默认的"平衡审查"输出形状

Step 4·RECONCILE（分类 findings，不橡皮图章）
  洞明对每个 finding 按优先级分类（第一个匹配的胜出）：
  1. 契约误读——finding 因契约不清，先修契约再重审
  2. 可执行——真问题，打回天权改
  3. 权衡——真问题但修复成本>接受成本，记录权衡
  4. 噪声——finding 错（审查者缺上下文），忽略并反思契约是否该补上下文
  ★ 不橡皮图章：洞明重新读代码核对每个 finding

Step 5·STOP（有界循环，不无限递归）
  停止条件（满足任一）：
  - 下一轮只有琐碎/已考虑的 findings
  - 已 3 轮（升级用户，不独自磨第4轮）
  - 用户说"放行"
  ★ 怀疑剧场自检信号：连续2+轮发现实质问题但零个分类为可执行
    = 洞明在验证而非怀疑，停止升级
```

**不照搬的部分**：agent-skills 原版支持跨模型升级（Gemini/Codex CLI 二审）。太一暂不内建——若需跨模型，用户可手动 `@瑶光` 走根因诊断（瑶光可调外部工具）。保持九星自洽，不引入外部模型依赖。

**只对非平凡交付触发**：机械操作（重命名/格式化/文件移动）、单行改动、明确指令无需 doubt 循环。怀疑每次按键则什么都交付不了。

### 11.2 九星通用：Rationalization Table（偷懒借口对照表）

**来源**：agent-skills + superpowers 共有的核心机制（两家都实测验证过）

**解决的问题**：LLM 会自我合理化——"这个太简单不需要走全流程""我快速实现一下就行""看起来没问题"。没有显式对照表，这些借口会悄悄生效。

**落地方式**：每颗星的 SKILL 末尾强制附 Rationalization Table。以下是九星的通用借口表（各星再补专属借口）：

| 借口（rationalization）| 事实反驳（reality）|
|----------------------|------------------|
| "这个太简单，不用走全流程" | 简单是主观判断。契约没说"简单可跳步"，就走完整流程 |
| "我能快速实现，不用盘点" | 快速实现恰恰是最该盘点的——开工前5分钟省的是后面的5小时返工 |
| "看起来没问题，应该可以了" | "看起来"不是验证。贴真实输出，跑过才算 |
| "这点小改动不会影响别处" | 改动影响面不由大小决定。登记 intervention.md 再说 |
| "审查者会很靠谱，这次就信任吧" | 信任与审查是两回事。独立审查不因信任而跳过 |
| "我上次就是这么做的，这次也行" | 上次的上下文和这次不同。按本次契约走，不按记忆走 |
| "用户在等，先交付再补验证" | 没验证的交付是负债。要么验证完交付，要么明确标注未验证 |
| "这个 finding 是审查者不懂，忽略" | 审查者缺上下文是信息，不是判决。重新读代码核对再分类 |

**各星专属借口补充**（示例，SKILL 落地时展开）：

| 星 | 专属借口 | 反驳 |
|----|---------|------|
| 天枢 | "用户说的很清楚，不用追问" | 清楚是幻觉。有歧义不追问=埋雷 |
| 天玑 | "这个 Task 不用标 mode，默认就行" | 不标 mode = 天权不知道走文心还是武毅 |
| 天权 | "白名单外那个文件也得改，顺手改了" | 顺手改=越权。登记 intervention.md，等司衡扩白名单 |
| 洞明 | "天权的自检报告说过了，我确认一下就行" | 确认=验证不是怀疑。走 doubt 循环，不读自检报告 |
| 瑶光 | "根因很明显，不用深挖" | 明显的根因常是症状。定位到具体星才算查清 |

### 11.3 天玑契约升级：耐久契约设计（AGENT-BRIEF 规则）

**来源**：skills 的 AGENT-BRIEF.md（仓库里最值得研究的契约设计文档）

**解决的问题**：契约可能因代码变化而腐烂——引用了文件路径/行号，文件一重构契约就失效；写了"怎么实现"而非"做什么"，绑死了实现方式。

**落地的四条契约纪律**（写进天玑 SKILL + contract-template.md）：

```
天玑落契约时，必须遵守耐久契约四纪律：

纪律一·耐久优于精确（Durability over precision）
  契约可能在 task-queue 里躺几天几周，代码会变。
  ✅ 描述接口签名、类型定义、行为契约
  ❌ 引用文件路径（会失效）
  ❌ 引用行号（会漂移）
  ❌ 假设当前实现结构不变
  
  好："SkillConfig 类型应接受可选 schedule 字段，类型为 CronExpression"
  坏："打开 src/types/skill.ts 第42行加个 schedule 字段"

纪律二·行为描述，非过程描述（Behavioral, not procedural）
  描述系统"该做什么"，不描述"怎么实现"。
  天权会自己探索代码库做实现决策，契约不该绑死。
  
  好："用户无参运行 /triage 时，看到需要分诊的 issue 摘要"
  坏："在主 handler 函数里加个 switch 语句"

纪律三·验收标准完整且独立可验证（Complete acceptance criteria）
  每条 DONE 必须独立可验证（grep/test/curl 能判定）。
  
  好："运行 gh issue list --label needs-triage 返回已分诊的 issue"
  坏："分诊应该正常工作"

纪律四·显式声明边界（Explicit scope boundaries）
  明确写出 Out of scope，防止天权镀金/越界。
  
  Out of scope 示例：
  - 不修改 1024 字符限制本身
  - 不支持多行 description
```

**与现有契约模板的关系**：这四条不是替代现有契约三要素（TASK-ID+DONE+DON'T），而是**强化 DONE 和 DON'T 的写法纪律**——
- 纪律一二约束 DONE 的写法（行为描述、禁路径行号）
- 纪律三强化 DONE 的可机检性（已有，现在有理论支撑）
- 纪律四强化 DON'T（已有 Out of scope 概念，现在显式化）

**反模式对照**（写进 contract-template.md 末尾）：

```markdown
## 反模式（天玑禁忌）

❌ 引用文件路径和行号
   坏："修改 src/auth/login.ts 第42行"
   好："LoginRequest 类型应增加 mfaCode 可选字段"

❌ 过程描述
   坏："在 handleSubmit 里加个 if 判断"
   好："提交时若 mfaRequired 为真，必须校验 mfaCode"

❌ 模糊验收
   坏："登录功能正常工作"
   好："POST /api/login 返回 200 + JWT；3次失败锁定"

❌ 无边界声明
   坏：（不写 Out of scope）
   好："Out of scope: 不改 user 表结构；不做第三方登录"
```

## 十二、文档架构

> 三类文档（运行态/产物态/索引态）+ 按 Epic 分目录 + 统一 frontmatter + 洞明自动归档 + INDEX/GUIDE 索引。

### 12.1 三类文档

| 类别 | 特征 | 谁写 | 生命周期 |
|------|------|------|---------|
| **运行态** | 随时变，司衡/星读写 | 司衡/各星 | 进行中存在，Epic归档时清理 |
| **产物态** | 完成后固化 | 各星产出 | 永久保留，原地不动 |
| **索引态** | 自动维护 | 洞明归档时更新 | 随归档更新 |

### 12.2 目录结构

```
docs/taiyi-school/
├── shared/                          # 运行态
│   ├── status.md                    # 当前状态（司衡写，各星读）
│   ├── task-queue.md                # 优先级队列（天玑写/司衡调度/洞明关）
│   └── intervention.md              # 溢出问题（各星写/洞明清点）
│
├── epics/                           # 产物态·按Epic组织
│   └── {epic-name}/
│       ├── requirements/            # 天枢产出
│       │   └── 需求真言.md
│       ├── designs/                 # 天璇产出
│       │   ├── 设计包.md
│       │   └── adr/
│       │       └── ADR-001-xxx.md
│       ├── contracts/               # 天玑产出
│       │   └── 契约-{TASK-ID}.md
│       ├── reports/                 # 各星报告
│       │   ├── 实现-{TASK-ID}.md    # 天权/开阳
│       │   ├── 自检-{TASK-ID}.md    # 玉衡
│       │   ├── 审查-{TASK-ID}.md    # 洞明
│       │   ├── 风险-{TASK-ID}.md    # 隐元（如有）
│       │   └── 诊断-{bug-id}.md     # 瑶光（如有）
│       └── archive/                 # Epic归档
│           └── {epic-name}.md       # 归档摘要
│
├── misc/                            # 杂项（不归属任何Epic的独立操作）
│   └── reports/
│
├── INDEX.md                         # 索引态·全局索引
└── GUIDE.md                         # 索引态·使用指南
```

### 12.3 Epic 谁定义

**Epic 由天枢在判需求时命名**（不是天玑拆解层级）。天枢产出需求真言时携带 epic 标识（小写中划线），后续所有文档继承。

Epic 不是"拆解动作"，是"文档组织单元"——一组相关 Task 及其文档的容器。天玑还是只拆 Task（两层），但每个 Task 知道自己属于哪个 epic。

**明确需求直接进天璇时**（绕过天枢），epic 由天璇命名。

### 12.4 运行态文件格式

**status.md（单值覆盖 + 挂起区）**：
```markdown
# 太一状态

| 字段 | 值 |
|------|-----|
| epic | {当前Epic} |
| current_star | {当前星} |
| current_task | {当前TASK-ID} |
| last_update | {ISO时间} |

## 挂起（Preempted）
| TASK-ID | epic | star | priority | 挂起原因 | wip分支 | 挂起时间 |

## 阶段日志（>50条截断保留最近20）
- [{时间}] {星} {动作} {结果}
```

**task-queue.md（优先级队列）**：
```markdown
# 任务队列

## 进行中（Running）
| TASK-ID | epic | 描述 | star | priority | mode |

## 就绪（Ready，按priority排序）
| TASK-ID | epic | 描述 | mode | priority | 依赖 |

## 挂起（Preempted）
| TASK-ID | epic | star | priority | 挂起原因 | wip分支 | 挂起时间 |

## 完成（Done）
| TASK-ID | commit | 完成时间 |
```

**intervention.md（溢出问题）**：
```markdown
# 溢出问题

## Open
| INT-ID | 类型 | 来源 | 描述 | 严重度 | 发现时间 |

## Closed（Epic归档时移入该Epic archive）
| INT-ID | 去向 | 关闭时间 |
```

### 12.5 产物态文档统一格式

所有产物文档用统一 YAML frontmatter + Markdown body：
```markdown
---
type: requirement | design | contract | report | diagnosis
task_id: TASK-xxx（如适用）
epic: {epic名}
star: {产出星}
status: draft | done | approved
created: YYYY-MM-DD
upstream: [依赖的上游文档]
downstream: [下游消费方]
---

# {文档标题}

{正文}
```

**不是每个角色一套格式，而是按文档类型一套模板**。角色只是 frontmatter 的 `star` 字段值。这让任何星读到文档立刻知道"这是什么、谁产的、完没完成、我该不该处理"（LLM 友好四原则的落地）。

### 12.6 归档机制（洞明执行）

**触发**：Epic 下所有 Task 完成 + 洞明最终审查通过 → 洞明执行归档。

**归档动作**：
```
1. 产物态文档原地不动（不移动，路径不变——耐久性）
2. 在 epics/{epic}/archive/ 写归档摘要：
   - Epic名 + 完成时间
   - Task清单（ID/描述/commit）
   - 产出文档清单（路径）
   - 遗留intervention（如有）
   - 经验教训（如有）
3. 清理shared层（见12.7）
4. 更新INDEX.md
5. status.md → epic: IDLE
```

**归档不移动文件**——移动会让引用路径失效。原地保留，索引指向。

### 12.7 shared 层归档（防无限膨胀）

shared 三个文件分两种生命周期：

| 文件 | 会膨胀吗 | 归档方式 |
|------|---------|---------|
| status.md | ❌（单值覆盖）| 阶段日志>50条截断保留最近20，不归档 |
| task-queue.md | ⚠ 完成区膨胀 | Epic归档时，该Epic的完成Task移入archive |
| intervention.md | ⚠ closed区膨胀 | Epic归档时，该Epic的closed INT移入archive |

**以 Epic 完成为归档单位**——不在 Epic 间间隙清理（防割裂）。归档后 shared 只保留当前活跃信息，历史在对应 Epic 的 archive 里。

**misc 清理**：misc/reports/ 的独立报告 6 个月未引用 → 洞明清理（TTL 兜底）。

### 12.8 INDEX.md（全局索引·洞明归档时更新）

```markdown
# 太一索引

## Epic 索引
| Epic | 状态 | 完成时间 | 归档路径 | Task数 |

## ADR 索引
| ADR | 决策 | Epic | 路径 |

## intervention 历史
| INT-ID | 类型 | 去向 | 关闭时间 |
```

### 12.9 GUIDE.md（使用指南）

```markdown
# 太一指南

## 目录结构说明（各目录职责）
## 文档命名规范（{类型}-{TASK-ID}.md / ADR-{序号}-{slug}.md）
## 归档规则（Epic完成→洞明归档→原地保留→索引指向）
## 司衡队列读写规则
  - 天玑写 task-queue（新增Task）
  - 司衡读 task-queue（调度就绪Task）
  - 洞明改 task-queue（标记完成）
  - 其他星只读不写
```

### 12.10 关键设计决策

1. **产物文档按 Epic 分目录**（同一Task的文档在一起，方便追溯）
2. **归档不移动文件**（路径不变，索引指向）
3. **intervention 放 shared**（可能跨 Epic，全局管理）
4. **status/task-queue/intervention 是唯一运行态文件**（司衡无状态，从这三文件读全局状态）
5. **统一 frontmatter**（不按角色分格式，按文档类型分模板）
