# 太一学派 · taiyi-school

> 天文同构的 AI Agent 角色分工协作体系。

太一学派把软件开发拆成九星（平级执行节点）+ 司衡（调度中枢），每颗星单一职责，靠文档接力协作。结构源头是**天文同构**——北斗九星围绕北极星（契约）旋转，司衡（帝车）调度推进。

**脱胎于[鲁班学派](https://github.com/zhanglilong23/luban-school)，汲取其工程内核（契约/审查/归档），去修仙装饰，以天文同构为结构根基。**

---

## 一、为什么需要太一学派

LLM 单 Agent 写小项目好用，写大项目会失控——上下文丢失、职责混乱、没验证就完工、问题遗失。太一学派用**角色分工 + 契约驱动 + 文档接力**解决这些痛点：

- **上下文会丢** → 每星独立会话，靠文档交接不靠记忆
- **声称"做完了"** → 门禁可机检（grep/test/curl 级）
- **默默填空** → 天枢浮假设、天璇浮技术假设
- **问题遗失** → 溢出三道闭环（登记 + 清点 + 复看）
- **3 次失败空转** → 熔断 + 质疑架构
- **凭空设计** → 天璇探索现有代码先行

## 二、角色体系（九星 + 司衡）

| 星 | 别称 | 职责 | 入口 |
|----|------|------|------|
| **天枢** | 贪狼·指极星 | 判需求（真伪/边界/价值）| ✅ @天枢 |
| **天璇** | 巨门 | 设计 + 判 Epic 边界 | ✅ @天璇 |
| **天玑** | 禄存 | 拆任务（两层结构）| ❌ 仅司衡调度 |
| **天权** | 文曲 | 文心编码（重质量）| ❌ |
| **玉衡** | 廉贞 | 自检（内化进天权/开阳）| ❌ |
| **开阳** | 武曲 | 武毅攻坚（重效率）| ❌ |
| **瑶光** | 破军 | 查根因（双面诊断）| ✅ @瑶光 |
| **洞明** | 左辅（隐）| 质门守门 + 归档 | ✅ @洞明 |
| **隐元** | 右弼（隐）| 非功能性风险守护 | ❌ |
| **司衡** | 帝车 | 调度中枢（纯只读路由）| ✅ @司衡（主入口）|

**双入口**：
- `@司衡` — 流水线模式，自动续接
- `@星宿` — 单星直达，不续接

## 三、三层职责分离（核心设计）

```
需求层（天枢）       Epic 层（天璇）        Task 层（天玑）
requirements/   →   epics/{epic}/      →   epics/{epic}/tasks/
REQ-NNN.md           design.md              TASK-NNN.md
```

**一个需求 → 多个 Epic**（如"商城系统"拆为 mall-mvp / mall-payment / mall-management），每个 Epic 独立可上线，支持多分支并行。

## 四、快速开始

### 安装（Claude Code 插件）

```bash
# 在你的项目根目录
git clone https://github.com/zhanglilong23/taiyi-school.git .taiyi-school
# 或作为 Claude Code 插件安装（详见 .claude-plugin/）
```

### 初始化文档骨架

```
/taiyi-school:setup
```
或手动运行 setup SKILL 的步骤。

### 启动一个需求

```
# 流水线模式（推荐）
@司衡 帮我做一个用户管理功能

# 或单星直达
@天枢 我想做一个用户管理功能
```

### 维护项目查 bug

```
@瑶光 支付接口报错了
```

## 五、文档架构（两层分离）

```
docs/taiyi-school/
├── requirements/          需求层（天枢产出，进 git）
├── _workspace/            工作层（分支本地，.gitignore）
├── epics/                 Epic 层（天璇判定 + 各星产出，进 git）
├── diagnoses/             瑶光独立诊断
├── misc/                  杂项 + reviews/
├── INDEX.md               顶层导航
└── GUIDE.md               使用指南
```

**关键**：`_workspace/` 是分支本地的当前态（不进 git，无合并冲突），产物态进 git 靠目录结构天然隔离。详见 [docs/taiyi-school/GUIDE.md](docs/taiyi-school/GUIDE.md)。

## 六、与鲁班学派的对比

| 维度 | 鲁班 | 太一 |
|------|------|------|
| 角色数 | 5 角色 + 2 法器 | 9 星 + 司衡（职责单一）|
| 结构源头 | 修仙位格（装饰）| 天文同构（同构）|
| 需求方法论 | 问道三阶 | 天枢四步（多正判）|
| 设计角色 | 无（司南兼）| 天璇（独立 + 判 Epic）|
| 拆解层级 | EPIC/TASK/STORY/SPEC 四层 | Task + 依赖两层（反官僚）|
| 自主权 | 二档（STORY/SPEC）| 三档（锁死/约束/放权）|
| 调度器 | 周天仪 + 巡天仪（两个）| 司衡（单一纯只读路由）|
| 状态文件 | shared/（进 git，有合并问题）| _workspace/（.gitignore，无合并）|
| 风险守护 | 玄鉴兼 | 隐元（五维度独立）|
| 部署 | 墨子 | 不覆盖（归运维）|
| 工程化 | 完整（插件/hooks/多Agent）| 基础+（plugin/hooks/setup + 版本管理 + 静态测试 + 方法论增强）|

**一句话**：太一在方法论与架构设计上全面超越鲁班，鲁班在工程化成熟度上仍有优势。

## 七、核心文件

| 文件 | 作用 |
|------|------|
| `CONTEXT.md` | 领域术语 + 文档架构 + 偏离说明（先读这个）|
| `docs/history/DESIGN-PROPOSAL.md` | 完整设计方案（历史归档稿，落地以 CONTEXT.md 偏离说明为准）|
| `SKILL-AUTHORING.md` | SKILL 写作规范 |
| `skills/<星>/SKILL.md` | 十星技能定义（九星 + 司衡）|
| `skills/<星>/references/` | 星专属资源（产出模板）|
| `skills/<星>/references/` | 星自带产出模板（技能自包含，多星共用模板各存副本）|
| `docs/taiyi-school/GUIDE.md` | 文档体系使用指南 |
| `.version-bump.json` | 版本字段声明（所有 manifest 的单一真源）|
| `scripts/bump-version.sh` | 版本一致性管理脚本（bump/check 双模式）|
| `tests/run-all.sh` | 静态测试套入口（frontmatter/references/司衡只读/版本/方法论术语）|
| `docs/ENGINEERING.md` | 工程化说明（版本管理/测试/方法论增强依据/hooks 展望）|

## 八、设计原则

1. **天文同构不是比喻**——命名基于真实星象含义，改名会破坏内在逻辑
2. **反官僚**——两层结构（非四层）、默认放权、玉衡/隐元内化
3. **文档接力**——靠 frontmatter 的 upstream/downstream，不靠对话记忆
4. **门禁可机检**——HARD-GATE 是 grep/test 级机械检查
5. **诚实边界**——不假装提供做不到的能力（无看门狗、无自动部署）
