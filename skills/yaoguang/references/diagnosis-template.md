# 诊断报告模板（瑶光产出）

> 双面诊断：照骨（bug 根因）+ 观势（架构体检）。
> 被动触发（洞明打回）路由回源头星；主动召唤（@瑶光）只出报告不续接。
> 原文件追加修改，不手动版本化。

---

## 用法一·照骨（bug 根因诊断）

> 归属某 Task 时用此格式。文件名 `diagnosis.md`，落 `epics/{epic}/reports/TASK-{NNN}/diagnosis.md`。
> 不归属任何 Epic 时，文件名 `BUG-{NNN}.md`，落 `diagnoses/`。

```markdown
---
type: diagnosis
task_id: TASK-{NNN}        # 或 bug_id: BUG-{NNN}（独立诊断时）
epic: {继承}               # 独立诊断时为 null
star: yaoguang
status: done
created: YYYY-MM-DD
upstream:
  - epics/{epic}/reports/TASK-{NNN}/review.md    # 洞明打回单（被动时）
downstream: [{路由目标星}]
---

# 诊断报告: TASK-{NNN} / BUG-{NNN}

## §A 症状描述

- 现象：{具体什么现象}
- 触发场景：{什么操作/输入下出现}
- 影响范围：{影响什么}

## §B 反馈回路（摄迹·核心）

<HARD-GATE>
先不读代码猜原因，先建一个能复现的信号。
没有反馈回路，盯代码再久也没用。
</HARD-GATE>

- 复现方式：{失败测试 / curl / CLI / 浏览器脚本 / trace / harness / fuzz / bisection}
- 复现命令：`{可复现的命令}`
- pass/fail 信号：{确定性信号}

## §C 坏味定性（寻脉）

| 坏味 | 是否命中 | 证据 |
|------|---------|------|
| 过度特化（只对某 case 生效）| | |
| 巨石回归（函数过大，改一处动全身）| | |
| 虚假测试（测试永远通过）| | |
| 教条实现（照搬模式不顾上下文）| | |
| bug复现（同类问题反复）| | |

## §D 根因（探源·因果链）

```
错误值从哪产生：{源头}
谁用了错误值调用这里：{调用链}
源头位置：{最终根因位置}
→ 在源头修，不在症状修
```

<HARD-GATE>
修复 3 次仍失败 → 不是假设错了，是架构错了。
停止修复，质疑架构，路由天璇。
</HARD-GATE>

## §E 凶险定级

- 定级：**{P0 / P1 / P2}**
- 理由：{为什么这个级别}

## §F 路由建议（精准路由，不直接修）

| 根因定位 | 路由到 | 理由 |
|---------|--------|------|
| **{实现偏差 / 架构问题 / 契约歧义 / 需求问题 / 非功能性风险}** | **{天权/开阳 / 天璇 / 天玑 / 天枢 / 隐元}** | {为什么} |

> 瑶光不改代码。定位根因 + 路由。

## §G 修复后验证

修复后如何确认修好了：
- 反馈回路变绿：`{复现命令}` → {预期 pass}
- 回归不破：`{测试命令}` → {预期全 pass}
```

## 用法二·观势（架构体检）

> Epic 完成时调度。结论写入对应 Epic 的归档摘要，或独立产出。
> 文件名 `assessment.md`，落 `epics/{epic}/reports/assessment.md`（Epic 级，不归属单 Task）。

```markdown
---
type: assessment
epic: {继承}
star: yaoguang
status: done
created: YYYY-MM-DD
upstream:
  - epics/{epic}/design.md
  - epics/{epic}/tasks/         # 全部 Task
downstream: [洞明归档决策]
---

# 架构体检: Epic {epic}

## §A 观势（模块边界 / 依赖图）

- 巨石/上帝类：{有 / 无，附位置}

## §C 寻脉（复杂度热点）

| 热点位置 | 问题 | 建议 |
|---------|------|------|

## §D 结论

- 接口偏离设计：{有 / 无}
- 复杂度热点：{有 / 无}
- 架构调整建议：{需回天璇 / 无需}
- 是否可归档：**{可归档 / 需回天璇重新评估}**
```

## 填写要点

- **照骨**：§B 反馈回路是核心——没有复现信号不猜测根因
- **照骨**：§D 根因必须追到源头，不在症状修
- **照骨**：§F 路由必须精准——区分实现/架构/契约/需求/非功能性
- **观势**：Epic 完成时调度，洞明归档前必须拿到观势结论
- **主动召唤**：只出报告不续接，用户决定下一步
