# 需求真言模板（天枢产出）

> 天枢判需求后产出，落 `.taiyi/requirements/REQ-{NNN}-{slug}.md`。
> 需求语义层，独立于 Epic（Epic 由天璇判定）。
> 原文件追加修改，不手动版本化（git log 留痕）。

```markdown
---
type: requirement
req_id: REQ-{NNN}
slug: {小写中划线}
star: tianshu
status: done
revision: 1
created: YYYY-MM-DD
last_modified: YYYY-MM-DD
epics: []                          # 天璇判定 Epic 后追加（如 [mall-mvp, mall-payment]）
upstream: []                       # 客户混沌，无文件
downstream: [天璇判 Epic 边界]
---

# 需求真言

**混沌原念**: {用户最初说的那句话，原样保留}

**我假设的（已确认/已纠正）**:
1. {假设1——标注用户确认了/纠正了}
2. {假设2}

**所求之道**: {凝练后的一句话，用户真正要什么}

**筑基之痛**: {具体痛点场景，可写进 Issue}

**真需求**: {最小可行需求范围}

**伪需求（已斩）**:
- {被排除的需求} —— {排除理由}

**Not Doing**:
- {看似相关但本次不做的} —— {为什么不做的理由}

**未明之境**: {还没想清楚的，留给下一轮}

**用户确认**: {用户是否确认此真言为其所求之道}

---

## 变更记录

> 修改时追加一行。原文件追加修改，不建 v2 文件。

| 版本 | 时间 | 修改者 | 修改原因 | 来源 |
|------|------|--------|---------|------|
| v1 | YYYY-MM-DD | 天枢 | 初版 | @天枢 探索 |
```

## 填写要点

- **混沌原念**：原样保留用户原话，不加工
- **假设**：天枢脑补的都列出，让用户确认/纠正（防默默填空）
- **伪需求已斩**：必须附排除理由（为什么不做）
- **Not Doing**：显式声明边界，防 scope creep
- **一句话原则**：所求之道必须能压缩成一句话，不能则没凝练完
- **变更记录**：bug 修正/需求调整时追加版本，不覆盖不重建文件

## 何时修改本文件

- 洞明打回 → 瑶光诊断 → 根因=需求问题 → 路由天枢重新正判
- 客户追加需求 → 天枢补充
- 发现伪需求未斩干净 → 天枢补斩

修改时：直接改正文 + frontmatter revision+1 + 变更记录追加一行。
